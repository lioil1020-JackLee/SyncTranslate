# SyncTranslate 優化建議

> 專案：SyncTranslate  
> 類型：Windows 桌面即時口譯工具  
> 目前主流程：音訊擷取 → 串流 ASR → LLM 翻譯 → TTS → 指定輸出裝置播放

---

## 1. 這份文件的目的

這份文件把兩件事整合在一起：

1. 架構與產品面的優化建議
2. 可直接執行的重構 TODO

目標不是再補幾個零碎功能，而是讓 SyncTranslate 一次走到「更穩、更可測、更好維護」的骨架。

---

## 2. 我對目前專案的整體判斷

SyncTranslate 已經不是 demo，而是很接近可持續演進的產品原型。從目前可見的模組劃分與 README 的描述來看，整體方向是正確的：

- 有明確的雙向翻譯模式與 session 概念
- 有把音訊、ASR、翻譯、TTS、字幕、狀態分成模組
- 有健康檢查、診斷、設定頁與 GUI 管理流程
- 有針對即時翻譯做 partial/final、sliding window、context 與避免重複播報等設計
- 有設定檔 fallback 與 legacy migration 的意識

但也出現了很典型的「原型成熟期」問題：

- 協調邏輯開始集中在 Router / UI
- 即時管線缺少明確 queue 邊界與背壓策略
- 錯誤、事件、診斷資料還不夠結構化
- prompt/模型策略可用，但還沒產品化
- 測試與 benchmark 不夠成體系

如果你現在想一次性修正到位，最值得投資的不是再堆功能，而是：

**把事件流、模組邊界、診斷、測試、翻譯策略正式化。**

---

## 3. 目前做得好的地方

### 3.1 模組切分方向正確
目前已經不是把所有流程塞進 GUI，而是有相對清楚的職責分工：`AudioRouter`、`ASRManager`、`TranslatorManager`、`TTSManager`、`TranscriptBuffer`、`StateManager` 等。這是非常好的基礎。

### 3.2 翻譯鏈路有即時場景意識
從現有設計描述來看，你不是把 LLM 當一般聊天模型在用，而是有明確考慮：

- partial / final
- fragment 翻譯
- context deque
- speak only on final
- CJK 長度與節奏
- 避免重複播報

這代表方向是對的。

### 3.3 設定相容性觀念正確
`settings.py` 已經有 fallback 與 migration，表示你有想到配置 schema 演進，而不是把 config 當一次性檔案。

### 3.4 使用者導向不錯
README 已經把系統需求、VB-CABLE 路由、健康檢查、快速開始、頁面用途講清楚，這對本地部署型工具非常重要。

## 4. 核心問題摘要

### 問題 A：協調邏輯偏集中
當前的架構雖然有分模組，但 session 啟停、路由、資料流協調仍偏集中。這會讓後續加功能時，某幾個檔案膨脹得很快。

### 問題 B：即時資料流沒有明確背壓策略
在即時口譯場景，任何一段變慢都會拖垮體驗。現在最需要的是：

- 哪裡排隊
- 哪裡丟舊 partial
- 哪裡保 final
- 哪裡可中斷
- 哪裡可降級

### 問題 C：partial/final 的生命週期還不夠正式
如果目前主要靠 source 或最新 partial 推測更新關係，後續在字幕穩定性、日誌追蹤、回放、匯出時會很痛。

### 問題 D：錯誤與診斷還不夠結構化
如果錯誤主要以字串 callback 流動，UI 能顯示，但很難做：

- 分級
- 聚合
- 統計
- 匯出
- 問題回溯

### 問題 E：翻譯策略還停留在單一 prompt 優化
你現在 prompt 方向是對的，但產品面更需要 profile，而不是只靠一段越來越長的 prompt。

### 問題 F：測試與 benchmark 不夠成套
這是所有多模組即時系統最後都會遇到的問題：

- 改 prompt 不知道有沒有退步
- 改 queue 不知道延遲有沒有變差
- 換模型不知道哪個更適合你的實際場景

---

## 5. 一次性修正到位的建議藍圖

我建議把重構分成三層：

### 第一層：把資料流正式化
核心是「事件模型 + queue 邊界 + lifecycle」。

### 第二層：把策略抽象化
核心是「translation profile + provider adapter + device profile」。

### 第三層：把驗證制度化
核心是「benchmark + regression + diagnostics report」。

這三層補齊之後，專案會從「很能跑的原型」變成「可持續優化的產品骨架」。

---

## 6. 詳細優化建議

## 6.1 把即時資料流改成顯式 pipeline + queue 邊界

### 建議
把目前概念上的流程：

`audio chunk -> ASR -> translation -> transcript -> TTS`

改成正式事件流。

建議的事件模型：

- `AudioChunkEvent`
- `ASRPartialEvent`
- `ASRFinalEvent`
- `TranslationPartialEvent`
- `TranslationFinalEvent`
- `TTSTask`
- `DiagnosticsEvent`
- `ErrorEvent`

每個 stage 要有自己的 queue 與策略：

- ASR queue：保留最新，舊 partial 可丟
- Translation queue：同 source 同 utterance 僅保留最新 partial
- TTS queue：只播 final； backlog 過長時可丟過時任務
- Diagnostics queue：只保留最近 N 筆

### 收益
- 背壓更可控
- local / remote 更容易隔離
- latency 更容易量測
- 未來換模型或後端更輕鬆

---

## 6.2 把 AudioRouter 瘦身成 façade

### 建議
目前 `AudioRouter` 很適合保留成 UI 單一入口，但內部應下沉拆分：

- `SessionOrchestrator`：mode、生命週期、啟停
- `RealtimePipeline`：local / remote 兩條處理鏈
- `RouterStatsCollector`：metrics 聚合
- `PipelineRegistry`：source 啟用與關閉

`AudioRouter` 只保留 façade 角色。

### 收益
- 單檔複雜度下降
- 後續功能不會全部擠進 Router
- 更好測試與 mock

---

## 6.3 為 partial / final 建立正式的版本與升級規則

### 建議
所有 ASR 與翻譯事件都帶：

- `utterance_id`
- `revision`
- `source`
- `stage`
- `created_at`
- `text`

規則：

- partial 只能更新同一 `utterance_id`
- final 直接升級並取代 partial
- `TranscriptBuffer` 不再只靠「刪除最新 partial」推測
- TTS 只接受 final

### 收益
- 字幕穩定度大幅提升
- partial/final 關係可追蹤
- 匯出、錄製、回放、分析都更容易

---

## 6.4 錯誤處理升級成結構化事件

### 建議
不要只靠 `on_error(str)`。

定義：

```python
@dataclass(slots=True)
class ErrorEvent:
    level: str
    module: str
    source: str | None
    code: str
    message: str
    detail: str | None
    created_at: float
```

所有模組只送 event，由 UI、logger、diagnostics panel 訂閱。

### 收益
- 可分 warning / error / fatal
- 更容易聚合與匯出
- 使用者回報問題時資訊更完整

---

## 6.5 翻譯策略改成 profile，不要只靠單一 prompt

### 建議
建立 `translation_profile`：

- `live_caption_fast`
- `live_caption_stable`
- `speech_output_natural`
- `technical_meeting`

每個 profile 控制：

- prompt template
- context 長度
- partial 觸發門檻
- max tokens
- 術語保守程度
- 是否允許語氣自然化
- 是否允許輕度補主語

### 收益
- 比單一 prompt 更可維護
- 不同模型更容易調校
- 可以更像產品設定，而不是工程參數

---

## 6.6 建立 LLM provider adapter 層

### 建議
不要只把 `ollama` / `lm_studio` 寫成 if/else 分支。

抽出 provider capability：

- `supports_json_schema`
- `supports_chat_messages`
- `supports_temperature`
- `supports_keep_alive`
- `supports_prefill`
- `supports_response_format`

不同 backend 各自 adapter。

### 收益
- 後端差異被隔離
- client 不會越長越亂
- 以後加 vLLM / OpenAI-compatible server / SGLang 比較乾淨

---

## 6.7 加 translation cache 與 prefix-aware 去重

### 建議
針對 partial 場景，加入兩層 cache：

1. exact cache：完全相同直接回傳
2. prefix-aware cache：只有在字串增量達門檻時才重翻

### 收益
- 降低 LLM 呼叫次數
- 降低 GPU 負載
- 降低字幕抖動

---

## 6.8 把 TTS queue 做成可中斷、可丟棄、可排序

### 建議
TTS queue 至少支援：

- partial 不播
- final 才播
- 新 final 可取消舊未播任務
- 長句分段播放
- backlog 過長時優先保最新 final

### 收益
- 降低「翻譯早就過時但還在講」的問題
- 即時口譯體驗大幅提升

---

## 6.9 將 device profile 正式化

### 建議
不要把 VB-CABLE 的假設散落在 UI、README、常數與 config。

建立：

- `vb_cable_default`
- `voicemeeter_default`
- `custom_manual`

由 resolver 統一決定裝置 mapping。

### 收益
- 降低使用者部署摩擦
- 減少音訊裝置名稱差異造成的支援成本

---

## 6.10 建立 benchmark、golden tests 與 diagnostics report

### 建議
建立三種測試集：

1. 音訊回歸集
2. 翻譯回歸集
3. 端到端 session 回歸集

指標至少包含：

- ASR latency p50 / p95
- Translation latency p50 / p95
- TTS backlog
- dropped partial count
- final delivery time
- JSON compliance

session 結束時輸出 `session_report.json`：

- config snapshot
- selected devices
- model/backend info
- source enablement
- avg latency
- queue stats
- recent errors

### 收益
- 後續所有改動都有依據
- 更容易比較模型、prompt、參數
- 更容易 debug 使用者環境

## 8. 我建議的實際執行順序

### 第 1 波：低風險高報酬
先做這 4 件：

1. `utterance_id` / `revision`
2. 結構化 `ErrorEvent`
3. translation cache
4. TTS queue 取消與優先級

這一波會直接提升穩定性與使用體感，而且不需要立刻大搬風。

### 第 2 波：核心架構升級
接著做：

1. `AudioRouter` façade 化
2. pipeline 與 queue 邊界正式化
3. provider adapter
4. translation profiles

這一波是最大的架構收益。

### 第 3 波：產品化與可回歸性
最後做：

1. benchmark suite
2. session diagnostics report
3. integration/golden tests
4. UI ViewModel 化

這一波會決定專案後續能不能持續快改而不壞。

## 9. 如果你想真的一次修到位，最少要守住的原則

### 原則 1：不要一邊重構一邊擴需求
重構期間先不要再加新功能，例如新模型、新 TTS、新 UI 頁面。否則會一直偏航。

### 原則 2：先定事件模型，再改模組
不要先拆檔案。先把事件流定義清楚，再做模組重構。

### 原則 3：先保行為一致，再求設計漂亮
重構初期先確保三個模式都能跑，再逐步優化抽象。

### 原則 4：所有重構都要有驗證入口
至少要有：

- 1 個 fake pipeline integration test
- 1 份 benchmark corpus
- 1 份 session report

## 10. 可以直接開 issue 的題目

### 架構
- Refactor AudioRouter into SessionOrchestrator and RealtimePipeline
- Introduce typed event model for ASR, Translation, TTS, and Diagnostics
- Add utterance_id-based partial/final reconciliation

### 翻譯
- Add translation profiles for live caption and speech output
- Add exact and prefix-aware translation cache
- Split backend capabilities into provider adapters

### 音訊 / TTS
- Add cancellable prioritized TTS queue
- Introduce device profiles and resolver

### 測試 / 診斷
- Export session diagnostics snapshot on stop
- Add benchmark corpus for zh/en real-time interpretation
- Add golden tests for translation JSON compliance
- Add integration tests with fake ASR / translator / TTS

## 11. 結論

如果只用一句話總結：

**SyncTranslate 現在最需要的不是更多模型，而是更正式的事件流、背壓策略、策略抽象、診斷與回歸機制。**

你已經有一個方向正確、功能完整、很接近產品的原型。現在最值得做的是把它從「能跑」升級成「可持續優化」。

把這份文件裡的 Phase 1 到 Phase 4 做完，整個專案的穩定性與可維護性就會明顯上升；把 Phase 5 到 Phase 7 補完，才真正會進入產品級迭代節奏。
