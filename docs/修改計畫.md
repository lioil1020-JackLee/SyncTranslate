# 修改計畫

> 實作狀態：已完成第一版落地與驗收。ASR v3 已加入 final confidence gate、同模型 rescue、中文 fallback router、UI 準確度模式、diagnostics effective params、regression corpus metadata、manifest 範本、missing/duplicate/hallucination/final latency 指標與對應測試。2026-05-11 已新增 curated stable regression corpus（20 中文、10 英文、5 中英混合），並完成 meeting/dialogue 35/35 baseline 與 packaged onedir smoke。

本計畫目標是把 SyncTranslate 的 ASR 從目前約 85-90% 的辨識表現，升級成更穩定、可量測、可回歸測試的 v3 ASR 架構。核心建議一句話是：

> 不要只靠單一模型或固定參數硬撐辨識率，而是建立「正式素材集 + 信心分數判斷 + 低信心 final 救援 + 中文模型路由」的 ASR v3 流程。

## 目標

- 提升中文、英文與 auto 模式下的 final 辨識穩定度。
- 減少 miss sentence、錯斷句、重複句、幻覺文字。
- 保持 partial 字幕低延遲，避免所有句子都被高成本重跑。
- 建立固定測試素材與報告，讓 ASR 調參不再只依賴畫面感覺。
- 讓使用者可以在低延遲、平衡、高準確模式之間切換。

## 設計原則

- partial 仍走快速路徑，不阻塞即時顯示。
- final 才做品質判斷與必要時的救援辨識。
- 只重跑低信心句子，不對所有句子做雙模型或高 beam。
- 原始 ASR 文字保留忠實輸出，標點整理與翻譯前清理應維持可選。
- 所有參數變更都必須能被 regression corpus 驗證。

### 素材類型

建立 `downloads/asr_corpus/` 或外部資料夾作為本機素材來源。若素材版權不適合公開，僅提交 manifest 範本與說明，不提交音檔本體。

建議至少收集以下類型：

- 新聞播報：語速穩定、標準國語、句子較長。
- 會議討論：多人聲、停頓多、插話、口語詞。
- 小說旁白：長句、情緒語氣、專有名詞。
- 雙人對話：短句密集、語氣詞多、快速來回。
- 短影音：背景音樂、音效、語速快、音質不穩。
- 雜訊場景：風扇聲、鍵盤聲、遠距收音、低音量。
- 男女聲混合：避免 profile 只對單一聲線最佳化。

### 每筆素材需要的檔案

每個 sample 建議包含：

- `audio.wav` 或 `audio.flac`：16 kHz mono 最佳，若來源是 stereo 也保留原始版本。
- `reference.txt`：人工校正逐字稿。
- `metadata.yaml`：素材描述。

`metadata.yaml` 建議欄位：

```yaml
id: zh_news_001
language: zh-TW
category: news
duration_sec: 120
speaker_count: 1
gender: mixed
noise_level: low
speech_rate: normal
has_music: false
reference_quality: human_verified
notes: 台灣新聞播報，句子較長。
```

### 評分指標

每次 ASR 改動都應輸出以下指標：

- CER：中文主要指標。
- WER：英文與空白分詞語言主要指標。
- punctuation-insensitive CER：忽略標點後的辨識率。
- missing sentence rate：漏句比例。
- duplicate rate：重複文字比例。
- hallucination rate：低信心或疑似幻覺比例。
- average final latency：final 平均延遲。
- dropped chunks：是否因負載過高丟音訊。

### 驗收標準

第一階段完成標準：

- 至少 20 筆中文素材、10 筆英文素材、5 筆混合語言素材。
- 每個主要類型至少 2 筆素材。
- regression runner 可以輸出 JSON 與 Markdown 報告。
- 任一 ASR 參數改動後，可以比較 baseline 與新結果。

## 第二階段：低信心 final 救援辨識

### 核心流程

目前 partial 與 final 仍維持現有流程。新增 final rescue 後，流程改為：

```text
audio segment
  -> fast final decode
  -> confidence check
  -> high confidence: accept
  -> low confidence: rescue decode
  -> candidate selection
  -> emit final transcript
```

### 低信心判斷條件

可使用 faster-whisper metadata 與文字規則判斷：

- `avg_logprob` 過低。
- `no_speech_prob` 過高但文字仍被產生。
- `compression_ratio` 過高，疑似重複或幻覺。
- final 文字過短，但音訊片段不短。
- final 文字與前一 final 重疊過高。
- 句子只包含語氣詞、單字或明顯錯語言內容。

### rescue 策略

低信心才重跑，最多 1-2 次：

- 提高 `final_beam_size`。
- 使用較長 final audio context。
- 對中文素材嘗試不同 `condition_on_previous_text`。
- 調整 `no_speech_threshold`。
- 使用較保守的 temperature fallback。

### candidate selection

候選結果評分可以考慮：

- metadata 信心分數。
- 是否符合目標語言。
- 與前後句的重複程度。
- 文字長度是否合理。
- 是否包含異常 loop。
- 是否比原結果有更低 missing/duplicate 風險。

### 延遲策略

- partial 不等待 rescue。
- high-confidence final 不增加延遲。
- low-confidence final 可先顯示初版，再於背景救援成功後修正。
- UI 可用設定切換是否允許背景修正 final。

## 第三階段：中文模型路由與 fallback

### 目標

中文素材不固定只依賴 belle-zh-ct2 或 large-v3-turbo，而是依據模式、語言、信心分數與負載選擇模型。

### 建議策略

- 平衡模式：使用目前預設模型，低信心才 fallback。
- 高準確模式：中文 final 可優先使用較準確設定，低信心時使用另一模型救援。
- 低延遲模式：關閉雙模型 fallback，只保留基本 confidence filter。

### fallback 觸發條件

- 中文連續句子 CER 在 regression 中偏高的類型。
- 當前 final metadata 顯示低信心。
- 偵測到短句 miss 或錯語言。
- 使用者啟用高準確模式。

### 避免的做法

- 不要每一句都同時跑 belle 與 large-v3-turbo。
- 不要在 queue 壓力高時啟動高成本 fallback。
- 不要讓 fallback 阻塞 partial 顯示。

## 第四階段：斷句與 endpoint 強化

### 可改善項目

- 中文模式下對短 pause 更保守，避免一句被切太碎。
- 對超長 segment 做安全切分，避免 final 太長導致翻譯品質下降。
- final rescue 時允許帶前後 context，但輸出時保留主句。
- 對前後句重疊文字做更嚴格去重。
- 對疑似漏句場景保留短音訊 tail，避免下一句開頭被吃掉。

### 驗收指標

- duplicate rate 下降。
- missing sentence rate 下降。
- 平均 final latency 不明顯惡化。
- regression corpus 中會議與對話素材表現改善。

## 第五階段：UI 與設定

### 新增 ASR 準確度模式

建議在設定頁加入：

- 低延遲：partial 優先，不做 final rescue 或雙模型 fallback。
- 平衡：低信心 final 才救援。
- 高準確：允許更高 beam、較長 context、中文 fallback。

### 診斷頁顯示 effective params

診斷頁應顯示目前實際生效狀態：

- ASR 模型。
- ASR 語言。
- endpoint profile。
- beam / final beam。
- VAD 參數。
- partial interval。
- final history。
- queue 壓力。
- dropped chunks。
- final priority 是否啟用。
- rescue 是否啟用。
- fallback 模型是否啟用。

## 分階段落地順序

### Phase 1：素材與量測

- 建立 corpus 目錄規格。
- 補 regression manifest 範本。
- 產生 baseline 報告。
- 確認目前 85-90% 的真實來源是模型錯誤、斷句錯誤、字幕標準錯誤，還是標點差異。

### Phase 2：final rescue

- 新增低信心判斷。
- 新增同模型不同參數重跑。
- 新增候選結果選擇器。
- 在平衡模式預設啟用。

### Phase 3：中文模型 fallback

- 新增中文模型路由。
- 只在低信心或高準確模式啟用。
- 加入 queue 壓力保護，避免 GPU 過載。

### Phase 4：UI 與文件

- 加入 ASR 準確度模式。
- 加入 effective params 診斷顯示。
- 更新所有文件與測試說明。

## 預期成效

- 單純微調參數：預期提升約 1-3%。
- final rescue + 更穩斷句：預期提升約 3-6%。
- 加入中文 fallback 與正式 corpus 調參：特定素材有機會穩定超過 90-93%。

實際上限仍取決於音訊品質、語速、背景噪音、字幕標準與模型本身能力。v3 的重點不是承諾所有素材都達到固定百分比，而是讓專案能用可量測方式持續逼近更好的 ASR 表現。
