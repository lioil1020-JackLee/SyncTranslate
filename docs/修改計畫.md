### P0-1 daemon 線程未捕獲異常

**問題描述**

`ASRWorkerV2._run()` 在 daemon 線程中執行，任何未預期的例外將導致線程靜默終止，而主程序完全不知道 ASR 功能已停止運作。同樣的問題存在於 `TranslationDispatcher._run()`。

**影響文件**

- `app/infra/asr/worker_v2.py`（`start()` 方法，`_run()` 方法）
- `app/application/translation_dispatcher.py`（`_run()` 方法）

**當前程式碼（worker_v2.py）**

```python
def start(self, on_event: Callable[[V2RuntimeEvent], None]) -> None:
    self.stop()
    self._on_event = on_event
    self._stop_event.clear()
    self._thread = Thread(target=self._run, daemon=True)
    self._thread.start()
```

`_run()` 內部沒有頂層 `try/except`，如果 backend 初始化失敗或 numpy 操作出現例外，線程直接終止。

**修改方案**

**步驟 1**：在 `worker_v2.py` 的 `_run()` 方法外層加入頂層例外捕獲與重試邏輯：

```python
def _run(self) -> None:
    import logging
    _log = logging.getLogger(__name__)
    try:
        self._run_inner()
    except Exception:
        _log.exception("ASRWorkerV2 線程意外崩潰，ASR 功能已停止")
        # 通知外部監控（可選：呼叫 on_event 傳遞錯誤事件）
        if self._on_event:
            try:
                self._on_event(V2RuntimeEvent(
                    text="[ASR 線程已崩潰，請重啟]",
                    is_final=True,
                    is_early_final=False,
                    start_ms=0,
                    end_ms=0,
                    latency_ms=0,
                ))
            except Exception:
                pass
```

**步驟 2**：在 `translation_dispatcher.py` 的 `_run()` 方法外層加入例外記錄：

```python
def _run(self) -> None:
    import logging
    _log = logging.getLogger(__name__)
    try:
        self._run_inner()
    except Exception:
        _log.exception("TranslationDispatcher 線程意外崩潰")
```

**步驟 3**：新增全域 `threading.excepthook`，在 `main.py` 或 `app_factory.py` 啟動時安裝：

```python
import threading
import logging

def _thread_excepthook(args: threading.ExceptHookArgs) -> None:
    if args.exc_type is SystemExit:
        return
    logging.getLogger("threading").critical(
        "Unhandled exception in thread %s",
        args.thread.name if args.thread else "<unknown>",
        exc_info=(args.exc_type, args.exc_value, args.exc_traceback),
    )

threading.excepthook = _thread_excepthook
```

**驗收標準**

- [x] 在 `_run()` 中人工拋出例外，確認錯誤訊息出現在日誌中
- [x] 主程序不因線程崩潰而靜默失效
- [x] `threading.excepthook` 在應用啟動時即安裝

### P0-2 translation_dispatcher 靜默吞嚥例外

**問題描述**

`TranslationDispatcher._process()` 在呼叫 `self._event_processor(event)` 時捕獲所有例外並直接 `pass`，導致翻譯管道中的錯誤完全無法被偵測。當 `_event_processor` 為 `None` 時，底層翻譯邏輯也有類似問題。

**影響文件**

- `app/application/translation_dispatcher.py`（`_process()` 方法，約第 137-145 行）

**當前程式碼**

```python
def _process(self, event: ASREventWithSource) -> None:
    if self._event_processor is not None:
        try:
            self._event_processor(event)
        except Exception:  # noqa: BLE001
            pass  # errors should be handled by the caller's event_processor
        return
```

**修改方案**

將靜默 `pass` 改為記錄例外，保留原本不讓例外往上傳播的設計意圖（避免 worker 線程崩潰），但確保可觀測性：

```python
import logging
_log = logging.getLogger(__name__)

def _process(self, event: ASREventWithSource) -> None:
    if self._event_processor is not None:
        try:
            self._event_processor(event)
        except Exception:  # noqa: BLE001
            _log.warning(
                "event_processor raised an exception for event source=%r text=%r",
                getattr(event, "source", "?"),
                getattr(event, "text", "?")[:60],
                exc_info=True,
            )
        return
    # 後續翻譯邏輯同樣加入 warning 日誌
    try:
        ...
    except Exception:  # noqa: BLE001
        _log.warning("Built-in translation failed for event source=%r", ..., exc_info=True)
```

**驗收標準**

- [x] 人工讓 `event_processor` 拋出例外，確認 warning 出現在日誌
- [x] worker 線程不因此崩潰
- [x] 不影響正常翻譯流程的效能

### P0-3 worker_v2 stop 未清理 _segment_chunks

**問題描述**

`_segment_chunks` 是 `list[np.ndarray]`，存放當前語音段落的所有音頻區塊。雖然 `stop()` 方法已存在，但若長時間連續語音或意外未觸發 finalize，該 list 可能無限增長，導致記憶體溢出。

**影響文件**

- `app/infra/asr/worker_v2.py`（`__init__` 中 `_segment_chunks`、`stop()` 方法）

**當前程式碼（stop 方法中已有）**

```python
def stop(self) -> None:
    self._stop_event.set()
    if self._thread and self._thread.is_alive():
        self._thread.join(timeout=8.0)
    self._thread = None
    self._endpointing.reset()
    self._frontend.reset()
    self._segment_chunks = []  # ✅ stop 時已清理
```

**實際問題**：`_segment_chunks` 在 `_run_inner()` 的主迴圈中累積，**沒有大小上限**。若 ASR backend 速度跟不上或 finalize 被延遲，單一 segment 可積累數百個 chunk（每個 chunk 約 4KB @ 16kHz 100ms），長時間運行可達 GB 級。

**修改方案**

**步驟 1**：在 `__init__` 加入段落最大長度限制常數：

```python
# 最大段落時長：60 秒 @ 16kHz，100ms/chunk = 600 個 chunk
_SEGMENT_MAX_CHUNKS = 600
```

**步驟 2**：在 `_accumulate_chunk()` 或加入段落 chunk 時檢查上限：

```python
def _append_to_segment(self, chunk: np.ndarray, sample_rate: float) -> None:
    self._segment_chunks.append(chunk)
    self._segment_sample_rate = sample_rate
    if len(self._segment_chunks) > _SEGMENT_MAX_CHUNKS:
        # 強制截斷：丟棄最早的 chunk，避免記憶體溢出
        self._segment_chunks = self._segment_chunks[-_SEGMENT_MAX_CHUNKS:]
        self._debug("segment_chunks truncated: exceeded max limit")
```

**步驟 3**：在段落完成（finalize）後立即釋放：

```python
def _finalize_segment(self) -> None:
    # ... 原有 finalize 邏輯
    self._segment_chunks = []  # 確保立即釋放
```

**驗收標準**

- [x] 長時間（>60 秒）單一語音段落不會讓 `_segment_chunks` 無限增長
- [x] 截斷時有 debug 日誌
- [x] 截斷後 ASR 繼續正常工作（可能有短暫的品質下降）

### P0-4 PowerShell device_name 命令注入風險

**問題描述**

`SystemDeviceVolumeController._set_endpoint_volume()` 將 `device_name`（來自配置文件）直接作為 PowerShell 命令列參數傳入，未限制字符集。惡意配置可能透過注入特殊字符執行任意 PowerShell 指令。

雖然配置文件通常由本機使用者管理，但作為防禦性設計原則，應做輸入驗證。

**影響文件**

- `app/infra/audio/device_volume_controller.py`（`_set_endpoint_volume()`，約第 30-55 行）

**當前程式碼**

```python
device_name = canonical_device_name(selector).strip()
if not device_name:
    return
command = [
    "powershell", "-NoProfile", "-ExecutionPolicy", "Bypass", "-File",
    str(self._script_path), "-Flow", flow,
    "-DeviceName", device_name,  # 直接使用，無字符驗證
    "-VolumePercent", str(int(round(clamped * 100.0))),
]
```

**修改方案**

在傳入命令前，對 `device_name` 進行字符白名單驗證：

```python
import re

# 允許：Unicode 字母、數字、空格、括號、連字號、底線、點號
_SAFE_DEVICE_NAME_RE = re.compile(r'^[\w\s\-().（）【】\[\]]+$', re.UNICODE)
_SAFE_FLOW_VALUES = frozenset({"capture", "render"})

def _validate_device_name(name: str) -> str:
    """驗證設備名稱僅含安全字符，防止命令注入。"""
    if not name:
        return name
    if not _SAFE_DEVICE_NAME_RE.match(name):
        # 記錄警告但不直接拋出，讓功能降級而非崩潰
        import logging
        logging.getLogger(__name__).warning(
            "Device name contains potentially unsafe characters, skipping volume control: %r", name
        )
        return ""
    return name

def _set_endpoint_volume(self, *, flow: str, selector: str, scalar: float) -> None:
    if sys.platform != "win32":
        raise RuntimeError("Real device volume control is only supported on Windows.")
    # 驗證 flow 參數
    if flow not in _SAFE_FLOW_VALUES:
        raise ValueError(f"Invalid flow value: {flow!r}")
    device_name = _validate_device_name(canonical_device_name(selector).strip())
    if not device_name:
        return
    ...
```

**驗收標準**

- [x] 含有 `; rm -rf /` 等特殊字符的 device_name 被過濾，不執行命令
- [x] 正常設備名稱（含中文、空格、括號）仍可正常運作
- [x] 驗證失敗時有 warning 日誌，程序不崩潰

### P0-5 SessionState.FAILED 後無法自動恢復

**問題描述**

`SessionController.start()` 在 `AudioRouter.start()` 失敗時，將狀態設為 `SessionState.FAILED` 後沒有任何恢復機制。使用者必須重啟整個應用程式才能再次嘗試，使用者體驗極差。

**影響文件**

- `app/application/session_service.py`（`start()` 方法，約第 55-75 行）

**當前程式碼**

```python
except Exception as exc:
    try:
        self._audio_router.stop()
    except Exception:
        pass
    with self._lock:
        self._state = SessionState.FAILED
        self._last_failure = str(exc)
    return SessionResult(ok=False, message=str(exc))
```

**修改方案**

**步驟 1**：在失敗後改為 IDLE 而非 FAILED，允許使用者重試：

```python
except Exception as exc:
    try:
        self._audio_router.stop()
    except Exception:
        pass
    with self._lock:
        # 改為 IDLE，讓使用者可以重試；保留 last_failure 供診斷
        self._state = SessionState.IDLE
        self._last_failure = str(exc)
    return SessionResult(ok=False, message=str(exc))
```

**步驟 2**：若希望保留 FAILED 狀態用於區分不可恢復的嚴重錯誤，加入手動重置方法：

```python
def reset_failed(self) -> SessionResult:
    """在 FAILED 狀態下允許手動重置為 IDLE 以便重試。"""
    with self._lock:
        if self._state != SessionState.FAILED:
            return SessionResult(ok=False, message=f"Not in FAILED state (current: {self._state.value})")
        self._state = SessionState.IDLE
        self._last_failure = ""
    return SessionResult(ok=True, message="Session reset to IDLE")
```

**步驟 3**：在 UI 層面，FAILED 狀態顯示「重試」按鈕，呼叫 `reset_failed()` 後再次 `start()`。

**驗收標準**

- [x] 啟動失敗後，呼叫 `reset_failed()` 可恢復為 IDLE
- [x] 再次呼叫 `start()` 可正常啟動
- [x] FAILED 狀態的 `last_failure` 訊息清楚說明失敗原因

### P1-1 TTS 線程 stop 邏輯不完善

**問題描述**

`TTSManager.stop()` 中使用 `max(0.1, float(wait_timeout))` 作為 join 超時，但當 `wait_timeout=5.0` 時每個 join 都等待 5 秒，共 6 個線程最多等待 30 秒。更重要的是，synth worker 可能正在等待 TTS 合成網路請求，無法提前終止。

**影響文件**

- `app/infra/tts/playback_queue.py`（`stop()` 方法，約第 165-195 行）

**修改方案**

**步驟 1**：stop 前先通知各 worker 盡快退出（除了已有的 `_stop_event`，加入喚醒機制）：

```python
def stop(self, wait_timeout: float = 5.0) -> bool:
    self._stop_event.set()
    self._passthrough_stop.set()
    # 喚醒所有在 queue_changed Condition 等待的線程
    with self._queue_changed:
        self._pending.clear()
        self._ready_audio["local"].clear()
        self._ready_audio["remote"].clear()
        self._passthrough_pending["local"].clear()
        self._passthrough_pending["remote"].clear()
        self._queue_changed.notify_all()
    ...
```

**步驟 2**：改為平行 join 而非串行，縮短總等待時間：

```python
import concurrent.futures

all_workers = [
    *[w for w in self._synth_workers.values() if w],
    *[w for w in self._play_workers.values() if w],
    *[w for w in self._passthrough_workers.values() if w],
]
deadline = time.monotonic() + max(1.0, float(wait_timeout))
for worker in all_workers:
    remaining = max(0.0, deadline - time.monotonic())
    if worker.is_alive():
        worker.join(timeout=remaining)
```

**步驟 3**：若 join 後線程仍存活，記錄警告而非靜默忽略：

```python
still_alive = [w.name for w in all_workers if w and w.is_alive()]
if still_alive:
    import logging
    logging.getLogger(__name__).warning("TTS workers did not stop in time: %s", still_alive)
    stopped = False
```

**驗收標準**

- [x] `stop()` 在正常情況下 5 秒內完成（平行等待）
- [x] 線程未能停止時有 warning 日誌
- [x] 不影響正常 TTS 播放功能

### P1-2 子進程全面強制 PYTHONUTF8

**問題描述**

在繁體中文 Windows（CP950）環境，任何未強制 UTF-8 的子進程都可能因編碼問題崩潰。`healthcheck_service.py` 已正確加入 `PYTHONUTF8=1`，但其他啟動子進程的地方（如外部 runtime 啟動、benchmark 工具）未必如此。

**影響文件**

- `app/bootstrap/external_runtime.py`
- `tools/asr_benchmark/run_multi_benchmark.py`（如有子進程啟動）
- 任何呼叫 `subprocess.run()` 或 `subprocess.Popen()` 的地方

**修改方案**

**步驟 1**：建立統一的子進程輔助函數於 `app/infra/subprocess_utils.py`（若不存在）：

```python
"""統一的子進程啟動工具，強制 UTF-8 環境。"""
from __future__ import annotations
import os
import subprocess
import sys


def safe_subprocess_env(extra: dict[str, str] | None = None) -> dict[str, str]:
    """回傳確保 UTF-8 的環境變數字典。"""
    env = os.environ.copy()
    env.setdefault("PYTHONUTF8", "1")
    env.setdefault("PYTHONIOENCODING", "utf-8")
    if extra:
        env.update(extra)
    return env
```

**步驟 2**：搜尋所有 `subprocess.run(` 和 `subprocess.Popen(` 呼叫，確認都傳入 `env=safe_subprocess_env()`。

**步驟 3**：在 `external_runtime.py` 的 runtime 啟動處套用：

```python
from app.infra.subprocess_utils import safe_subprocess_env

proc = subprocess.Popen(
    cmd,
    env=safe_subprocess_env(),
    ...
)
```

### P1-8 競態條件 can_accept_asr 雙重 tick 鎖

**問題描述**

`StateManager.can_accept_asr()` 先呼叫 `self.tick()`（內部取鎖），然後再取鎖讀取 `_asr_enabled`。兩次取鎖之間存在 TOCTOU（Time-of-Check-Time-of-Use）問題：`tick()` 可能修改 `_asr_enabled`，但隨後另一個執行緒也可能修改它。

**影響文件**

- `app/domain/runtime_state.py`（`can_accept_asr()`、`tick()`，約第 55-75 行）

**當前程式碼**

```python
def can_accept_asr(self, source: str) -> bool:
    self.tick()        # 內部取鎖，可能修改 _asr_enabled
    key = source if source in ("local", "remote") else "local"
    with self._lock:   # 再次取鎖，tick 更新可能被覆蓋
        return self._running and self._asr_enabled.get(key, False)
```

**修改方案**

將 `tick()` 邏輯內聯至 `can_accept_asr()`，在單一鎖的保護下完成：

```python
def can_accept_asr(self, source: str) -> bool:
    key = source if source in ("local", "remote") else "local"
    now = time.time()
    with self._lock:
        # 在鎖內執行 tick 邏輯，避免 TOCTOU
        self._tick_locked(now)
        return self._running and self._asr_enabled.get(key, False)

def _tick_locked(self, now: float) -> None:
    """在已持有 _lock 的情況下更新 ASR 啟用狀態。"""
    if self._local_echo_guard_enabled and not self._tts_busy["local"]:
        if now >= self._echo_guard_resume_until["local"]:
            self._asr_enabled["local"] = True
    if not self._tts_busy["remote"] and now >= self._echo_guard_resume_until["remote"]:
        self._asr_enabled["remote"] = True

def tick(self) -> None:
    now = time.time()
    with self._lock:
        self._tick_locked(now)
```

**驗收標準**

- [x] `can_accept_asr()` 只取一次鎖
- [x] `tick()` 行為不變（現有測試通過）
- [x] `snapshot()` 中呼叫 `self.tick()` 也改為 `_tick_locked()`

self._enabled_events: dict[str, Event] = {
