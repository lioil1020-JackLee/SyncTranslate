# 架構說明

## 核心流程

目前 runtime 主線由以下模組串起來：

- [`app/application/audio_router.py`](/e:/py/SyncTranslate/app/application/audio_router.py)
- [`app/bootstrap/dependency_container.py`](/e:/py/SyncTranslate/app/bootstrap/dependency_container.py)
- [`app/application/session_service.py`](/e:/py/SyncTranslate/app/application/session_service.py)

實際流程：

1. `SessionController.start()` 啟動雙向 router
2. `AudioInputManager` 提供兩個 source
   - `remote` -> `meeting_in`
   - `local` -> `microphone_in`
3. `AudioRouter` 接收 audio chunk
4. 先依輸出模式決定是否做直通
5. chunk 同時送入對應 ASR pipeline
6. ASR 結果更新 original panel
7. 若該方向是翻譯模式，送入 LLM
8. 翻譯結果更新 translated panel
9. 若輸出模式為 `tts`，翻譯文字送入 TTS queue

## Direction model

目前產品行為固定是：

- `bidirectional`

也就是遠端與本地兩個方向永遠同時存在，只是每個方向可各自切成 `直通` 或 `翻譯`。

## Source 與 Output 對應

| source | capture device | original panel | translated panel | output channel | output device |
| --- | --- | --- | --- | --- | --- |
| `remote` | `audio.meeting_in` | `meeting_original` | `meeting_translated` | `local` | `audio.speaker_out` |
| `local` | `audio.microphone_in` | `local_original` | `local_translated` | `remote` | `audio.meeting_out` |

補充：

- `remote` source 的輸出會播到本地端
- `local` source 的輸出會送回會議端

## Output modes

目前實際使用的模式是：

- `passthrough`
  - 原聲直通
- `tts`
  - 翻譯後以 TTS 輸出

UI 上已不再要求靠 `翻譯目標 = 無` 來間接切模式，而是直接用模式下拉選單控制。

## UI 狀態管理

`LiveCaptionPage` 目前已改成以 `channel = remote / local` 為中心的共用邏輯：

- `ASR 語言`
- `模式`
- `翻譯目標`
- `TTS 聲線`
- original / translated label
- panel status

這樣可以避免 remote/local 兩邊邏輯分叉，降低之後改 UI 時只修一半的風險。

另外，`模式` 的 round-trip 規則已固定為：

1. UI 選 `直通 / 翻譯`
2. 寫回 `runtime.remote_translation_enabled / runtime.local_translation_enabled`
3. 重新 `apply_config()` 時優先信任這兩個顯式旗標

也就是 `翻譯目標` 與 `TTS 聲線` 只負責翻譯模式下的內容，不再反向決定模式本身。

## 2026-03-30 TTS 流程補充

- ASR 只負責輸出辨識文字。
- LLM 只負責依據 ASR 文字產生翻譯文字。
- TTS 只負責朗讀最終翻譯文字，不再額外改寫 spoken text。
- TTS queue 採合成與播放分離設計，降低句與句之間的空白等待。
- `僅 partial 使用上下文` 開啟時，只讓 partial 翻譯參考前文；final 翻譯仍只看當句，避免同文不同譯。
