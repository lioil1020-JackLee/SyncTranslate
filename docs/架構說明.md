# 架構說明

## 總覽

SyncTranslate 目前的主線架構是：

- 音訊擷取與路由
- `ASR v2`
- 翻譯
- TTS / passthrough
- UI / diagnostics / export

ASR 已從舊的 `legacy whisper streaming heuristics` 轉向 `ASR v2`。目前實際運作的核心是：

- `app/infra/asr/factory.py`
- `app/infra/asr/manager_v2.py`
- `app/infra/asr/worker_v2.py`
- `app/infra/asr/backend_resolution.py`
- `app/infra/asr/backend_v2.py`
- `app/infra/asr/endpointing_v2.py`
- `app/infra/asr/faster_whisper_adapter.py`

## 音訊資料流

1. `AudioCapture` 取得本地 / 遠端音訊
2. `AudioRouter` 將 chunk 送往對應通道
3. `ASRManagerV2` 建立或重用該通道 runtime
4. `EndpointingRuntime` 根據 backend 與 VAD 設定判斷 `speech_started / soft_endpoint / hard_endpoint`
5. `SourceRuntimeV2` 決定 partial / final 發送時機
6. transcript 送往翻譯與 TTS
7. UI 顯示 original / translated 面板
8. diagnostics 與 session report 記錄實際執行資訊

## 通道模型

專案固定使用兩條邏輯通道：

- `local`
- `remote`

這兩條通道各自有：

- ASR 語言
- 翻譯目標
- TTS 聲音
- queue
- runtime stats

因此：

- `local` 與 `remote` 可以同時發話
- 兩條通道會各自維護自己的 ASR runtime
- UI 與 session report 也會分開記錄兩邊狀態

## ASR v2

### 1. Factory

`create_asr_manager()` 會根據 `runtime.asr_pipeline` 建立：

- `legacy`
- `v2`

目前預設是 `v2`。

### 2. 語言自動路由

`backend_resolution.py` 會根據每通道 ASR 語言決定 backend：

- `zh` / `zh-TW` / `zh-CN` / `cmn` / `yue` -> `faster_whisper_v2`
- 其他語言 -> `faster_whisper_v2`
- `auto` -> `faster_whisper_v2`
- `none` -> disabled

這個決策只在 runtime 建立時做一次，不會在 session 中熱切換。

### 3. Backend pair

`backend_v2.py` 會建立：

- partial backend
- final backend

目前規則：

- 中文通道
  - `faster_whisper_v2:partial`
  - `faster_whisper_v2:final`
- 非中文 / auto
  - `faster_whisper_v2:partial`
  - `faster_whisper_v2:final`

### 4. Endpointing

`endpointing_v2.py` 提供 `EndpointingRuntime`。

目前設計：

- faster-whisper 中文鏈路優先使用 `silero_vad`
- 非中文鏈路可使用 `silero_vad`
- 若沒有可用 neural VAD，仍會保留 RMS 型 fallback 判斷

輸出的重要訊號：

- `speech_started`
- `speech_active`
- `speech_ended`
- `soft_endpoint`
- `hard_endpoint`

### 5. Worker

`worker_v2.py` 負責：

- chunk queue
- backlog 控制
- partial / final 發送
- soft endpoint / hard endpoint finalize
- backend runtime stats

### 6. faster-whisper lazy init

`faster_whisper_adapter.py` 是目前 ASR 模型載入與 runtime 狀態回報的入口。

它處理：

- process-level singleton
- lazy init
- thread-safe init
- init failure cache
- 已下載本機快取路徑優先
- requested device -> effective device 正規化

這讓程式不需要在啟動時就把 faster-whisper 模型全部載入。

## 翻譯與 TTS

### 翻譯

翻譯層目前仍以 LM Studio 為主。

主要模組：

- `app/infra/translation/engine.py`
- `app/infra/translation/lm_studio_adapter.py`

### TTS

TTS 與 passthrough 由：

- `app/infra/tts/playback_queue.py`

負責 queue、取消、backoff、裝置播放。

近期調整重點：

- `passthrough_playback_failed` 已做節流 / backoff
- 避免裝置失敗時持續刷 log 與吃掉 CPU

## UI 與 diagnostics

### Local AI 頁

`LocalAiPage` 現在已對齊新架構：

- 不再強調手選 ASR engine
- 直接提示中文會走 faster-whisper，非中文會走 faster-whisper
- 提示 requested device 與 effective device 可能不同

### Live Caption 頁

`LiveCaptionPage` 仍由每通道 ASR 語言作為主要入口。

每通道的 ASR 語言選單現在會提示：

- 中文 -> faster-whisper + silero-vad
- 非中文 / auto -> faster-whisper
- `none` -> 停用該通道 ASR

### Runtime stats

目前 diagnostics / session report 會輸出：

- `resolved_backend`
- `resolved_language_family`
- `backend_resolution_reason`
- `device_effective`
- `model_init_mode`
- `init_failure`

這些欄位是理解「為什麼兩邊結果不同」的重要依據。

## 舊架構狀態

舊的 `legacy` ASR 路徑仍存在，但已凍結：

- 不再作為主要擴充點
- 不再繼續堆新功能
- 等 `ASR v2` 完成完整實測驗收後，會進行第二階段清理

## Phase 1-4 商品化擴充（2026-04-16）

### ASR 後處理層

#### TranscriptPostProcessor

`app/application/transcript_postprocessor.py`

在 ASR partial / final 事件進入 transcript buffer 前做文字後處理：

- **partial 穩定前綴**：若新 partial 縮短幅度超過 80% 且前綴重疊 < 60%，保留前一個 partial，避免字幕閃爍
- **text normalization**：trim、全形→半形、重複空白、標點間距
- **glossary 套用**：final 事件上套用自訂術語表

#### GlossaryStore

`app/domain/glossary.py`

- 支援 `substring` 與 `exact`（word boundary）兩種比對模式
- case-insensitive（可設定）
- `conservative=True` 時跳過 substring 模式，僅做 exact 替換
- 在 `TranscriptPostProcessor` 中自動套用於 final 文字

#### GlossaryLoader

`app/infra/config/glossary_loader.py`

- 從 YAML / JSON 檔載入術語表
- 永遠不拋例外，失敗時回傳空 `GlossaryStore`
- 由 `AudioRouter._rebuild_postprocessor()` 在 config 刷新時重載

### Structured Logging

`app/infra/logging/runtime_logger.py`

- 背景 worker thread，呼叫端 non-blocking
- `RuntimeLogRecord` 記錄：timestamp、module、source、event_type、utterance_id、revision、latency_ms、degradation_level
- 支援 jsonl / text 格式
- 透過 `init_runtime_logger()` / `get_runtime_logger()` 存取

### Metrics

`app/domain/metrics.py`

- `UtteranceLatency` / `PipelineMetrics` / `MetricsCollector`
- thread-safe

---

### StreamingPolicy

`app/infra/asr/streaming_policy.py`

從 `worker_v2.py` 抽離的 streaming 決策層：

- **StreamingContext**：封裝所有決策所需狀態（音訊長度、backlog、cooldown 等）
- **StreamingDecision**：決定 `emit_partial / emit_final / suppress_partial / reset_endpointing_after_final`
- **StreamingPolicy**：三段 degradation 策略
  - `NORMAL`：正常 partial interval
  - `CONGESTED`：延長 partial interval（`congested_partial_interval_multiplier`，預設 1.5×）
  - `DEGRADED`：壓制 partial，降低 ceiling threshold（`degraded_partial_interval_multiplier`，預設 3×）
  - 根據 queue backlog 與 dropped_chunks_total 決定降級

### Endpoint Profiles

`app/infra/asr/endpoint_profiles.py`

6 個內建 endpoint profile，透過 `runtime.asr_profile_local` / `asr_profile_remote` 選用：

| 名稱 | 適用場景 |
|---|---|
| `default` | 一般室內環境 |
| `meeting_room` | 遠端會議 / 廣播語音 |
| `headset` | 頭戴式麥克風近講 |
| `noisy_environment` | 嘈雜環境 |
| `max_accuracy` | 犧牲延遲換精度 |
| `low_latency` | 即時字幕要求較嚴 |

---

### 音訊前處理鏈

`app/infra/asr/audio_pipeline/`

`AudioFrontendChain` 將音訊前處理組合成可插拔的 stage 鏈：

| Stage | 說明 |
|---|---|
| `HighpassStage` | IIR pre-emphasis（DC drift 抑制） |
| `NoiseReductionStage` | 包裝 `AsrSpeechEnhancerV2`，強度可調 |
| `MusicSuppressionStage` | 音樂背景壓制 |
| `LoudnessStage` | 溫和 AGC（mild gain normalization） |

AEC 擴充點：`chain.insert(0, aec_stage)` 即可在前端插入 AEC stage。

`AudioFrontendChain.build_default(...)` 產生預設順序的鏈。每個 stage 都實作 `AudioProcessingStage` Protocol（`process` + `reset`）。

### AudioRouter 責任分離

Phase 4 將 AudioRouter 的大段責任抽離成獨立模組：

#### ASREventProcessor

`app/application/asr_event_processor.py`

- latency gate（`max_latency_ms`）
- 調用 `TranscriptPostProcessor`
- 將結果寫入 transcript buffer

#### TranslationDispatcher

`app/application/translation_dispatcher.py`

- 非同步翻譯 queue（bounded，`queue_maxsize`）
- tail-drop overflow 策略（佇列滿時丟棄最舊）
- `on_translated` / `on_skipped` callback
- 已修正：`stop()` 後清空殘留 sentinel，確保 `start()` 後 worker 不會立即退出

#### TTSDispatcher

`app/application/tts_dispatcher.py`

- 依 channel output mode 決定是否入隊 TTS
- `maybe_enqueue(TTSEnqueueRequest) → bool`

#### PipelineMetricsCollector

`app/application/pipeline_metrics.py`

- `UtteranceMilestones` 追蹤每句話各階段時間點
- `PipelineMetricsCollector`：thread-safe，`window_size` 可設
- `record_asr_partial / record_asr_final / record_llm_final / record_tts_enqueue / record_tts_playback`
- `recent_latencies()` 回傳最近 N 句的各段延遲（ms）

### UI Controllers

`app/ui/controllers/`

從 MainWindow 抽離的 controller 層（可獨立單元測試）：

| Controller | 職責 |
|---|---|
| `SessionActionController` | session start / stop 生命週期、狀態回報 |
| `LiveCaptionRefreshController` | transcript buffer polling、line cache diff、panel 更新 callback |
| `ConfigHotApplyController` | debounce 熱套用（mark_pending → tick → on_apply）、suspend/resume |
| `HealthcheckController` | healthcheck start / poll / status 回報 |

MainWindow 仍可保留自己的 QTimer 基礎架構；這些 controller 是可獨立測試的純 Python 物件，不依賴 Qt。

## 目前限制

- 雙通道同時中文發話時，如果 `effective device = cpu`，延遲仍可能明顯增加
- speaker diarization 仍是實驗性功能，預設關閉
- `runtime.asr_v2_backend` 目前屬相容欄位，實際 backend 仍以通道語言路由為準

## CUDA 依賴固定策略（2026-04-16）

ASR v2 的 CUDA 路徑依賴 `torch/torchaudio` wheel 正確性。
目前專案已在依賴層固定：
- Windows、Linux x86_64 使用 `torch==2.10.0+cu128`、`torchaudio==2.10.0+cu128`
- 來源固定為 `download.pytorch.org/whl/cu128`

這與 runtime 的 `requested device` / `effective device` 檢查互補：
- 依賴層先保證 CUDA wheel
- 執行層再確認實際是否成功落到 GPU

## ASR 設定語意（UI）

ASR 進階設定已從方向維度切換為語言家族維度：
- 中文（faster-whisper）
- 非中文（faster-whisper）

注意：`auto` 語言仍保留通道 fallback，避免既有專案在未明確指定語言時產生非預期行為。
