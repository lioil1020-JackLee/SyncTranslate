# 架構說明

## 總覽

SyncTranslate 目前的主線架構是：

- 音訊擷取與路由
- `ASR v2`
- 翻譯
- TTS / passthrough
- UI / diagnostics / export

ASR 已從舊的 `legacy whisper streaming heuristics` 轉向 `ASR v2`。目前實際運作的核心是：

- `app/infra/asr/factory.py`
- `app/infra/asr/manager_v2.py`
- `app/infra/asr/worker_v2.py`
- `app/infra/asr/backend_resolution.py`
- `app/infra/asr/backend_v2.py`
- `app/infra/asr/endpointing_v2.py`
- `app/infra/asr/funasr_registry.py`

## 音訊資料流

1. `AudioCapture` 取得本地 / 遠端音訊
2. `AudioRouter` 將 chunk 送往對應通道
3. `ASRManagerV2` 建立或重用該通道 runtime
4. `EndpointingRuntime` 根據 backend 與 VAD 設定判斷 `speech_started / soft_endpoint / hard_endpoint`
5. `SourceRuntimeV2` 決定 partial / final 發送時機
6. transcript 送往翻譯與 TTS
7. UI 顯示 original / translated 面板
8. diagnostics 與 session report 記錄實際執行資訊

## 通道模型

專案固定使用兩條邏輯通道：

- `local`
- `remote`

這兩條通道各自有：

- ASR 語言
- 翻譯目標
- TTS 聲音
- queue
- runtime stats

因此：

- `local` 與 `remote` 可以同時發話
- 兩條通道會各自維護自己的 ASR runtime
- UI 與 session report 也會分開記錄兩邊狀態

## ASR v2

### 1. Factory

`create_asr_manager()` 會根據 `runtime.asr_pipeline` 建立：

- `legacy`
- `v2`

目前預設是 `v2`。

### 2. 語言自動路由

`backend_resolution.py` 會根據每通道 ASR 語言決定 backend：

- `zh` / `zh-TW` / `zh-CN` / `cmn` / `yue` -> `funasr_v2`
- 其他語言 -> `faster_whisper_v2`
- `auto` -> `faster_whisper_v2`
- `none` -> disabled

這個決策只在 runtime 建立時做一次，不會在 session 中熱切換。

### 3. Backend pair

`backend_v2.py` 會建立：

- partial backend
- final backend

目前規則：

- 中文通道
  - `funasr_v2:partial`
  - `funasr_v2:final`
- 非中文 / auto
  - `faster_whisper_v2:partial`
  - `faster_whisper_v2:final`

### 4. Endpointing

`endpointing_v2.py` 提供 `EndpointingRuntime`。

目前設計：

- FunASR 中文鏈路優先使用 `fsmn_vad`
- 非中文鏈路可使用 `silero_vad`
- 若沒有可用 neural VAD，仍會保留 RMS 型 fallback 判斷

輸出的重要訊號：

- `speech_started`
- `speech_active`
- `speech_ended`
- `soft_endpoint`
- `hard_endpoint`

### 5. Worker

`worker_v2.py` 負責：

- chunk queue
- backlog 控制
- partial / final 發送
- soft endpoint / hard endpoint finalize
- backend runtime stats

### 6. FunASR lazy registry

`funasr_registry.py` 是目前中文 ASR 與 `fsmn_vad` 的共享模型入口。

它處理：

- process-level singleton
- lazy init
- thread-safe init
- init failure cache
- 已下載本機快取路徑優先
- requested device -> effective device 正規化

這讓程式不需要在啟動時就把 FunASR / FSMN-VAD 全部載入。

## 翻譯與 TTS

### 翻譯

翻譯層目前仍以 LM Studio 為主。

主要模組：

- `app/infra/translation/engine.py`
- `app/infra/translation/lm_studio_adapter.py`

### TTS

TTS 與 passthrough 由：

- `app/infra/tts/playback_queue.py`

負責 queue、取消、backoff、裝置播放。

近期調整重點：

- `passthrough_playback_failed` 已做節流 / backoff
- 避免裝置失敗時持續刷 log 與吃掉 CPU

## UI 與 diagnostics

### Local AI 頁

`LocalAiPage` 現在已對齊新架構：

- 不再強調手選 ASR engine
- 直接提示中文會走 FunASR，非中文會走 faster-whisper
- 提示 requested device 與 effective device 可能不同

### Live Caption 頁

`LiveCaptionPage` 仍由每通道 ASR 語言作為主要入口。

每通道的 ASR 語言選單現在會提示：

- 中文 -> FunASR + FSMN-VAD
- 非中文 / auto -> faster-whisper
- `none` -> 停用該通道 ASR

### Runtime stats

目前 diagnostics / session report 會輸出：

- `resolved_backend`
- `resolved_language_family`
- `backend_resolution_reason`
- `device_effective`
- `model_init_mode`
- `init_failure`

這些欄位是理解「為什麼兩邊結果不同」的重要依據。

## 舊架構狀態

舊的 `legacy` ASR 路徑仍存在，但已凍結：

- 不再作為主要擴充點
- 不再繼續堆新功能
- 等 `ASR v2` 完成完整實測驗收後，會進行第二階段清理

## 目前限制

- 雙通道同時中文發話時，如果 `effective device = cpu`，延遲仍可能明顯增加
- speaker diarization 仍是實驗性功能，預設關閉
- `runtime.asr_v2_backend` 目前屬相容欄位，實際 backend 仍以通道語言路由為準
