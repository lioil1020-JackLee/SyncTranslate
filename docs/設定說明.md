## 目前主要設定區塊

### `audio`

負責輸入 / 輸出裝置路由：

- `meeting_in`
- `microphone_in`
- `speaker_out`
- `meeting_out`

### `language`

描述本地與遠端的來源語言 / 翻譯目標語言。

### `asr_channels`

每通道 ASR 參數：

- `local`
- `remote`

常用欄位：

- `model`
- `device`
- `compute_type`
- `beam_size`
- `final_beam_size`
- `no_speech_threshold`
- `vad.*`
- `streaming.*`

注意：

- `model` 主要影響 `faster-whisper` 路徑
- 中文 FunASR 路徑不直接使用這裡的 model 名稱做 backend 選擇

### `llm_channels`

每通道翻譯參數。

### `tts`

共用 TTS 預設。

### `tts_channels`

每通道 TTS 覆寫。

### `runtime`

這是目前最重要的區塊。

## `runtime` 關鍵欄位

### ASR 主線

- `asr_pipeline`
  - 預設 `v2`
- `asr_v2_backend`
  - 目前主要是相容欄位
  - 實際 backend 仍由通道語言路由決定
- `asr_v2_endpointing`
  - endpointing backend 的設定入口

### 每通道語言入口

- `local_asr_language`
- `remote_asr_language`

這兩個欄位直接決定每通道實際使用的 ASR backend。

路由規則：

- `zh` / `zh-TW` / `zh-CN` / `cmn` / `yue` -> `funasr_v2`
- 其他語言 -> `faster_whisper_v2`
- `auto` -> `faster_whisper_v2`
- `none` -> disabled

## 目前建議理解方式

### 1. ASR backend 不再由 UI 的 engine 欄位主導

現在真正主導 backend 的是：

- `runtime.local_asr_language`
- `runtime.remote_asr_language`

### 2. `device` 是 requested device，不一定等於 effective device

即使 `config.yaml` 寫：

```yaml
device: cuda
```

實際執行時仍可能因為 Torch / CUDA 環境而回退為 `cpu`。  
要以 diagnostics / session report 中的：

- `device_effective`

為準。

### 3. 雙通道同時中文發話時，要特別注意 CPU bottleneck

如果：

- `local_asr_language = zh-TW`
- `remote_asr_language = zh-TW`
- `device_effective = cpu`

那麼即使邏輯上已避免共享模型互相污染，雙通道仍可能因吞吐量不足而延遲上升。

## 精簡外部 config

目前外部 config 以「保留有效設定」為主。  
不需要持久化的 runtime metadata、相容性衍生欄位、部分 legacy shared 欄位，會由程式在載入後補齊，而不是都寫回 `config.yaml`。

## 推薦流程

1. 在 `字幕頁` 決定每通道 ASR 語言
2. 在 `Local AI` 頁調整 device、VAD 與 queue
3. 用 diagnostics 確認：
   - `resolved_backend`
   - `device_effective`
   - `init_failure`
4. 再根據 session report 微調
