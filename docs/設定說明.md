## 目前主要設定區塊

### `audio`

負責輸入 / 輸出裝置路由：

- `meeting_in`
- `microphone_in`
- `speaker_out`
- `meeting_out`

### `language`

描述本地與遠端的來源語言 / 翻譯目標語言。

| 欄位 | 預設 | 說明 |
|---|---|---|
| `meeting_source` | `en` | 遠端（會議）來源語言 |
| `meeting_target` | `zh-TW` | 遠端翻譯目標語言 |
| `local_source` | `zh-TW` | 本地來源語言 |
| `local_target` | `en` | 本地翻譯目標語言 |

> 注意：config migration 會自動將舊版欄位名稱（`remote_translation_target` / `local_translation_target`）對應至新欄位（`meeting_target` / `local_target`）。

### `asr_channels`

每通道 ASR 參數：

- `local`
- `remote`

常用欄位：

- `model`
- `device`
- `compute_type`
- `beam_size`
- `final_beam_size`
- `no_speech_threshold`
- `vad.*`
- `streaming.*`

注意：

- `model` 主要影響 `faster-whisper` 路徑
- 中文 faster-whisper 路徑不直接使用這裡的 model 名稱做 backend 選擇
- `asr_channels.*` 是基礎值；實際 runtime 會再經過 `app/infra/asr/language_profiles.py` 依語言調整。若 `local_asr_language = zh-TW`，中文 profile 會把 `no_speech_threshold` 調到 `0.25`，並套用較保守的 VAD / endpointing 參數。

### `llm_channels`

每通道翻譯參數。

### `tts`

共用 TTS 預設。

### `tts_channels`

每通道 TTS 覆寫。

### `runtime`

這是目前最重要的區塊。

## `runtime` 關鍵欄位

### ASR 主線

- `asr_pipeline`
  - 預設 `v2`
- `asr_v2_backend`
  - 目前主要是相容欄位
  - 實際 backend 仍由通道語言路由決定
- `asr_v2_endpointing`
  - endpointing backend 的設定入口

### 每通道語言入口

- `local_asr_language`
- `remote_asr_language`

這兩個欄位直接決定每通道實際使用的 ASR backend。

路由規則：

- `zh` / `zh-TW` / `zh-CN` / `cmn` / `yue` -> `faster_whisper_v2`
- 其他語言 -> `faster_whisper_v2`
- `auto` -> `faster_whisper_v2`
- `none` -> disabled

辨識率最常見的問題是語言鎖錯。例如實際說中文但通道設定為 `en` / `th`，會讓 faster-whisper 使用錯誤語言提示，結果通常比 `auto` 或正確語言差很多。

### 即時行為

- `sample_rate`
- `chunk_ms`
- `asr_pre_roll_ms`
- `stable_partial_min_repeats`
- `partial_stability_max_delta_chars`
- `asr_partial_min_audio_ms`
- `asr_partial_interval_floor_ms`
- `llm_partial_interval_floor_ms`
- `early_final_enabled`

目前穩定取向預設：

- `stable_partial_min_repeats = 3`
- `partial_stability_max_delta_chars = 6`
- `asr_partial_min_audio_ms = 360`
- `early_final_enabled = false`

即時 ASR 目前建議：

- 中文長句 / 會議 / 影片音訊：`asr_profile_* = meeting_room`
- 英文短句語音輸入：保留 `meeting_room` 或 `default` 也可以；語言 profile 會在未明確選 `max_accuracy` / `low_latency` 時自動套用較適合英文短句的 `turn_taking`
- 不建議一律使用 `max_accuracy`；實測顯示過長 final window 可能放大尾端幻聽
- 若以辨識率優先，`max_pipeline_latency_ms` 建議至少 `6000`

- `asr_final_correction_enabled`

`asr_final_correction_enabled` 會呼叫 LM Studio 做 final-only 文字修正。這不是 ASR decoder 本身，預設建議保持關閉；在中文長故事實測中，LLM correction 可能過度改寫 ASR 文字。若本機 LLM 穩定可用、且你確認內容是短句語音輸入，再考慮開啟。

## 目前建議理解方式

### 1. 目前真正主導 runtime 的是 UI preset 與 runtime 欄位

YAML 仍是設定快照，但實際執行行為主要由 UI 切換快速模式後熱套用的 runtime 值主導。

目前最重要的是：

- `runtime.local_asr_language`
- `runtime.remote_asr_language`
- `runtime.asr_profile_local`
- `runtime.asr_profile_remote`
- `runtime.early_final_enabled`
- `runtime.stable_partial_min_repeats`
- `runtime.partial_stability_max_delta_chars`

內建快速模式：

- `超穩定會議字幕`
  - 對應 `meeting_room` 取向，較保守切段
- `低延遲對話（穩定 ASR）`
  - ASR 對應 `meeting_room`，LLM / TTS 使用低延遲對話設定
  - 建議中文雙向對話優先使用，避免 `turn_taking` 對中文長句造成過度切段
- `低延遲雙向對話`
  - 對應 `turn_taking` 取向，偏向短句快速回應
  - 適合短句問答與搶話明顯的情境，中文準確率可能低於穩定 ASR 模式

### 2. `device` 是 requested device，不一定等於 effective device

即使 `config.yaml` 寫：

```yaml
device: cuda
```

實際執行時仍可能因為 Torch / CUDA 環境而回退為 `cpu`。  
要以 diagnostics / session report 中的：

- `device_effective`

為準。

### 3. 雙通道同時中文發話時，要特別注意 CPU bottleneck

如果：

- `local_asr_language = zh-TW`
- `remote_asr_language = zh-TW`
- `device_effective = cpu`

那麼即使邏輯上已避免共享模型互相污染，雙通道仍可能因吞吐量不足而延遲上升。

## 精簡外部 config

目前外部 config 以「保留有效設定」為主。  
不需要持久化的 runtime metadata、相容性衍生欄位、部分 legacy shared 欄位，會由程式在載入後補齊，而不是都寫回 `config.yaml`。

## 推薦流程

1. 在 `字幕頁` 決定每通道 ASR 語言
2. 在 `Local AI` 頁調整 device、VAD 與 queue
3. 用 diagnostics 確認：
   - `resolved_backend`
   - `device_effective`
   - `init_failure`
4. 再根據 session report 微調

## Phase 1-4 新增 runtime 欄位

### ASR 後處理

| 欄位 | 預設 | 說明 |
|---|---|---|
| `enable_postprocessor` | `true` | 啟用 TranscriptPostProcessor |
| `enable_partial_stabilization` | `true` | partial 穩定前綴（避免字幕回縮閃爍） |
| `glossary_enabled` | `false` | 啟用術語表套用 |
| `glossary_path` | `''` | YAML / JSON 術語表路徑 |
| `glossary_apply_on_partial` | `false` | partial 文字是否套用術語表 |
| `glossary_apply_on_final` | `true` | final 文字是否套用術語表 |

術語表格式範例（YAML）：

```yaml
- pattern: "Whisper"
  replace: "Whisper"
  case_sensitive: false
  mode: substring
- pattern: "LLM"
  replace: "LLM"
  mode: exact
```

### Structured Logging

| 欄位 | 預設 | 說明 |
|---|---|---|
| `enable_structured_logging` | `true` | 啟用背景 runtime logger |
| `runtime_log_format` | `jsonl` | 輸出格式（`jsonl` 或 `text`） |

結構化日誌輸出位置：`logs/runtime_*.jsonl`

### ASR Endpoint Profiles

| 欄位 | 預設 | 說明 |
|---|---|---|
| `asr_profile_local` | `meeting_room` | 本地通道 endpoint profile |
| `asr_profile_remote` | `meeting_room` | 遠端通道 endpoint profile |

可用 profile 名稱：

- `default`：一般室內環境（推薦起點）
- `meeting_room`：長句字幕 / 會議監聽 / 穩定度優先
- `headset`：頭戴式麥克風近講
- `noisy_environment`：嘈雜環境
- `max_accuracy`：犧牲延遲換精度
- `low_latency`：即時字幕要求較嚴
- `turn_taking`：雙向短句對談 / 搶話頻繁時優先降低切段延遲

目前 diagnostics 也會顯示：

- `postprocessor.final.rejected_count`
- `postprocessor.final.last_rejection_reason`
- `endpoint_signal.pause_ms`
- `speech_started_count / soft_endpoint_count / hard_endpoint_count`

若你要另外準備一份「雙向對談版」設定，可直接參考：

- [config.conversation.yaml](/e:/py/SyncTranslate/config.conversation.yaml)

### Streaming Degradation Policy

| 欄位 | 預設 | 說明 |
|---|---|---|
| `degradation_policy_enabled` | `true` | 啟用 streaming 降級保護 |

降級觸發條件：queue backlog 過高或 dropped chunks 累積過多。降級後會壓制 partial 發送頻率，降低 CPU / 翻譯壓力。
