# SyncTranslate 修改計畫 v2（產品化重構版）

> 版本日期：2026-05-22  
> 目標：把 SyncTranslate 重構為可攜、可部署、可在一般 Windows 電腦穩定使用的本地即時字幕與雙向語音工具。  
> 範圍：本版只整理並實作 SyncTranslate 現有產品主線；不導入 whisper.cpp、Windows 內建 ASR、Google/Azure/雲端 ASR。  
> 核心 ASR：維持 `faster-whisper / CTranslate2` 作為唯一正式 ASR backend。

---

## 0. 本版決策摘要

本版 v2 以產品化為目標，不以「少改程式」為目標。凡是現有架構在模式語義、音訊格式、直通、driver/bridge、UI 或設定模型上有根本混淆，都直接重構。

### 0.1 明確排除項目

以下內容只屬於前期技術比較與理解，不列入本次修改計畫：

- `whisper.cpp` backend
- Windows 內建 ASR / Win + H / SAPI / Voice Typing
- Google Speech-to-Text / Azure Speech / Deepgram 等雲端 ASR
- 多 ASR backend plugin 架構
- 以雲端 API 替代本地 ASR

本版只做：

```text
Windows audio capture / loopback / virtual driver
→ faster-whisper ASR
→ local llama.cpp translation
→ Edge TTS / direct passthrough / UI transcript
```

---

## 1. 產品模式重新定義

SyncTranslate 正式分成兩個產品模式：

1. **會議監聽模式**
2. **雙向語音模式**

這兩個模式的音訊路由、UI、啟用裝置、TTS 行為、ASR/LLM 行為完全分離。

---

## 2. 會議監聽模式

### 2.1 定義

會議監聽模式是純字幕、翻譯、記錄模式。

此模式只從本機取得一個 ASR 音源，產生原文與翻譯字幕，不把聲音送回任何通訊軟體，不啟動虛擬音效裝置，不啟動 virtual mic，不啟動 virtual speaker，不需要 bridge，不檢查 driver。

### 2.2 會議音源

使用者可在 UI 中選擇其中一種音源：

```text
1. 系統輸入
   例：麥克風、Line In、USB audio input、桌面混音輸入

2. 系統輸出 loopback
   例：目前播放 Zoom/Teams/Meet 聲音的喇叭、耳機、HDMI audio、USB headset
```

### 2.3 會議模式資料流

```text
會議監聽模式 / 系統輸入：

[Physical Input Device]
  → AudioCapture
  → ASR branch: 48k/native → mono → 16k
  → faster-whisper
  → local LLM translation
  → TranscriptBuffer
  → UI 原文 / 翻譯 / 匯出
```

```text
會議監聽模式 / 系統輸出 loopback：

[Physical Output Device loopback]
  → WASAPI loopback capture
  → ASR branch: 48k/native → mono → 16k
  → faster-whisper
  → local LLM translation
  → TranscriptBuffer
  → UI 原文 / 翻譯 / 匯出
```

### 2.4 會議模式硬性規則

會議監聽模式必須符合以下規則：

- 不建立 `VirtualAudioBridgeClient`
- 不建立 `VirtualSpeakerSource`
- 不建立 `VirtualMicrophoneSink`
- 不呼叫 `ensure_virtual_audio_runtime_ready()`
- 不檢查 virtual audio driver
- 不要求 Windows Test Mode
- 不啟動 remote source
- 不啟動 remote sink
- 不啟動 TTS audio output
- 不送任何音訊到 `meeting_out`
- 不做 direct passthrough
- TTS 關閉不影響 ASR
- TTS 關閉不影響 LLM 翻譯
- TTS 關閉不影響字幕與記錄

### 2.5 會議模式 UI

會議監聽模式只顯示：

```text
[音源類型：系統輸入 / 系統輸出 loopback]
[音源裝置：裝置下拉選單]
[ASR 語言：中文 / 英文 / 日文 / 韓文 / 泰文]
[翻譯目標：中文 / 英文 / 日文 / 韓文 / 泰文]
[開始 / 停止]
[清空字幕]
```

字幕區只顯示兩欄：

```text
┌ 會議原文 ───────────────┐ ┌ 會議翻譯 ───────────────┐
│ ASR 原文                │ │ LLM 翻譯                │
└────────────────────────┘ └────────────────────────┘
```

會議監聽模式 UI 中不顯示：

- 遠端原文
- 遠端翻譯
- 本地送出
- Virtual Mic
- Virtual Speaker
- TTS voice
- direct passthrough
- 雙向語音方向設定

## 3. 雙向語音模式

### 3.1 定義

雙向語音模式是線上通話雙向口譯模式。

此模式使用虛擬音效裝置，讓通訊軟體的遠端聲音進入 SyncTranslate，也讓 SyncTranslate 可以把本地翻譯語音送回通訊軟體。

### 3.2 雙向語音模式資料流

```text
遠端 → 本地：

[Communication App Output]
  → SyncTranslate Virtual Speaker
  → VirtualSpeakerSource
  → ASR / direct decision
  → faster-whisper
  → local LLM translation
  → TTS or direct passthrough
  → Physical Speaker / Headphones
```

```text
本地 → 遠端：

[Physical Microphone]
  → AudioCapture
  → ASR / direct decision
  → faster-whisper
  → local LLM translation
  → TTS or direct passthrough
  → SyncTranslate Virtual Microphone
  → Communication App Input
```

### 3.3 雙向語音模式方向策略

每個方向都有獨立 output policy：

```yaml
dialogue:
  remote_to_local:
    output_policy: translated_tts | direct_passthrough | subtitle_only

  local_to_remote:
    output_policy: translated_tts | direct_passthrough | subtitle_only
```

三種 policy 的行為如下。

#### translated_tts

```text
音訊
  → ASR
  → LLM 翻譯
  → TTS
  → 對應輸出裝置
  → 原文字幕 / 翻譯字幕
```

#### direct_passthrough

```text
音訊
  → format adapter / limiter
  → 對應輸出裝置
```

此模式跳過：

- ASR
- LLM
- TTS
- transcript
- endpointing
- VAD
- ASR frontend
- TTS queue
- passthrough gate
- passthrough warmup
- passthrough batch
- passthrough backoff

#### subtitle_only

```text
音訊
  → ASR
  → LLM 翻譯
  → UI 字幕 / 記錄
```

此模式不輸出音訊。

### 3.4 TTS voice 與 output policy 的關係

UI 層可以保留「TTS 語音 = 無」的直覺操作，但 config 內部必須使用明確 policy。

規則如下：

```text
雙向語音模式：
  TTS voice = 有聲線
    → output_policy = translated_tts

  TTS voice = 無
    → output_policy = direct_passthrough
```

會議監聽模式完全不使用此規則，因為會議監聽模式沒有 TTS output，也沒有 direct passthrough。

## 4. 音訊格式產品化決策

### 4.1 結論

SyncTranslate 採用四個清楚分離的音訊 domain：

| Domain | 格式 | 用途 |
|---|---|---|
| Windows / driver / virtual endpoint | 48kHz / PCM16 / 2ch | 對外裝置與通訊軟體相容 |
| App internal audio frame | float32 / native channels / sample_rate metadata | DSP、routing、passthrough、resampling |
| ASR input | float32 / 16kHz / mono | faster-whisper / Whisper 模型入口 |
| TTS output boundary | float32 internal → 48kHz / PCM16 / 2ch boundary | 播放與 virtual mic 輸出 |

此設計同時滿足：

- Windows 音訊裝置相容性
- 雙向語音 direct passthrough 的 stereo 保真
- ASR 的 16kHz mono 模型需求
- TTS / DSP 的 float32 處理精度
- driver / bridge 的穩定 ABI

### 4.2 不是整個 app 都改成 int16 stereo

產品化後的正確設計是：

```text
外部 boundary：48kHz / PCM16 / 2ch
內部運算：float32 AudioFrame + 明確 sample_rate / channels metadata
ASR branch：16kHz / mono / float32
```

原因是：

- ASR、VAD、增益、limiter、resampling、TTS 混音都需要 float32 運算。
- int16 只適合做 PCM device boundary，不適合作為整個 app 的內部處理格式。
- direct passthrough 要保留 stereo 時，必須保留原始 channel layout，不能先被 ASR branch 轉成 mono。
- physical microphone 有些是 mono，有些是 stereo；產品行為必須依裝置能力處理，而不是假設所有 mic 都是 stereo。
- 現有 `AudioCapture` 目前強制開 1 channel，這是目前行為，不是產品化目標。

### 4.3 麥克風是否 stereo 的處理

產品化規則如下：

```text
如果麥克風支援 2ch：
  雙向語音 direct_passthrough 使用 2ch capture，保持 stereo。
  ASR branch 另外 downmix 成 mono。

如果麥克風只支援 1ch：
  direct_passthrough 在輸出 boundary duplicate 成 2ch。
  ASR branch 直接使用 mono 後 resample 成 16k。

如果使用者只需要 ASR，不需要 direct passthrough：
  capture 可用 mono，降低成本。
```

因此雙向語音模式不能把同一份 audio buffer 先轉成 16k mono 再拿去直通。必須在 source 進來時分流：

```text
AudioFrame 48k/native channels
  ├─ direct passthrough branch：保留 stereo / native channels
  └─ ASR branch：downmix mono + resample 16k
```

### 4.4 direct passthrough 不受 16k mono ASR 影響

雙向語音模式 TTS 關閉後的真正直通必須走獨立分支：

```text
input AudioFrame
  → PassthroughRouter
  → resample to output rate if needed
  → preserve stereo if source is stereo
  → duplicate mono to stereo if source is mono
  → soft limiter
  → PCM16 stereo boundary
  → sink
```

此分支不進入：

- `ASRManagerV2.submit()`
- `FasterWhisperEngine._prepare_audio()`
- `resample_audio(... target_rate=16000)`
- endpointing
- VAD
- transcript validator
- translation dispatcher
- TTS queue

### 4.5 ASR branch 仍固定 16k mono

faster-whisper / Whisper 的模型入口固定為 16kHz mono。產品化後不能把 48kHz stereo 直接送進 faster-whisper 當作準確率提升手段。

ASR branch 固定如下：

```text
AudioFrame 48k/native channels
  → channel collapse / downmix to mono
  → high quality resample 48k → 16k
  → float32 mono
  → faster-whisper
```

### 6.1 現況問題

目前 `AudioCapture._preferred_input_channels()` 固定回傳 `1`。這代表所有實體輸入裝置都被強制以 mono 開啟。

產品化後此行為改為依 session role 決定。

### 6.2 新增 channels_policy

```python

    STEREO = "stereo"
    NATIVE = "native"
    STEREO_OR_MONO = "stereo_or_mono"
```

### 6.3 policy 對應

| 使用情境 | channels_policy |
|---|---|
| 會議監聽 / ASR only / 系統輸入 | mono |
| 會議監聽 / 系統輸出 loopback | stereo_or_mono |
| 雙向語音 / local translated_tts | mono or stereo_or_mono |

### 6.5 capture rate 策略

```text

其次裝置 default samplerate

```

```python
def resolve_direction_state(direction: str) -> SourceState:
```

## 10. Direct Passthrough Router

### 10.1 新增模組

```text
app/application/passthrough_router.py
```

### 10.2 職責

`PassthroughRouter` 專門處理真正直通，與 TTS queue 完全分離。

### 10.3 行為

```text
AudioFrame
  → output format adapter
  → soft limiter
  → short jitter buffer
  → sink.push_passthrough()
```

### 10.4 禁止行為

direct passthrough 禁止使用：

- TTS synthesis queue
- `_passthrough_gate_rms`
- `_passthrough_warmup_until`
- `_passthrough_gate_until`
- `_passthrough_batch_wait_sec`
- `_passthrough_backoff_until`
- ASR VAD
- ASR endpointing
- transcript postprocessor
- translation dispatcher

### 10.5 limiter

```python
def soft_limiter(x: np.ndarray, threshold: float = 0.95) -> np.ndarray:
    y = np.asarray(x, dtype=np.float32)
    peak = float(np.max(np.abs(y))) if y.size else 0.0
    if peak <= threshold:
        return y
    return np.tanh(y / threshold) * threshold
```

### 10.6 direct passthrough output rules

| Input | Output |
|---|---|
| 48k stereo | 48k stereo |
| 48k mono | duplicate to 48k stereo |
| 44.1k stereo | resample to 48k stereo |
| 16k mono | resample to 48k stereo |
| >2ch | select L/R or downmix to stereo |

## 11. TTSManager 重構

### 11.1 目標

`TTSManager` 只負責文字轉語音與 TTS 播放，不再承擔真正直通。

### 11.2 output modes

TTS 合成失敗只影響該 TTS task，不得修改 direct passthrough 的狀態。

```python
self._passthrough_backoff_until[channel] = ...
```

## 15. Virtual audio driver / bridge 產品化

### 15.1 產品化 driver contract

Virtual audio endpoints 對外固定宣告：

```text
sample_rate = 48000
channels = 2
bit_depth = 16
format = PCM
```

### 15.2 driver / bridge boundary

App internal 使用 float32。

寫入 virtual mic / bridge boundary 時轉換為：

```text
float32 [-1, 1]
  → soft limiter
  → stereo adapter
  → int16 interleaved
  → driver / bridge
```

讀取 virtual speaker / bridge boundary 時轉換為：

```text
int16 stereo / driver PCM
  → float32 AudioFrame
  → routing
```

### 15.3 bridge protocol version

新增 protocol version 2：

```python
{
  "protocol_version": 2,
  "sample_rate": 48000,
  "channels": 2,
  "bit_depth": 16,
  "dtype": "int16",
  "frames": n,
  "data_b64": "..."
}
```

啟動時 handshake：

```text
app protocol_version == bridge protocol_version
driver format == app expected format
```

版本不一致時，阻止雙向語音模式啟動並顯示修復指引。

### 15.4 meeting mode no-driver

會議監聽模式成為 no-driver 功能。

任何 Windows 電腦只要能啟動 app，就能使用：

```text
系統輸入 ASR
系統輸出 loopback ASR
字幕
翻譯
匯出
```

雙向語音模式需要 virtual driver。

### 15.5 任意 Windows 電腦部署策略

產品化 release 分兩層：

```text
Meeting Monitor Edition:
  無 driver 依賴
  無 Test Mode
  立即可用

Dialogue Voice Edition:
  需要 production-signed virtual audio driver
  installer 負責安裝 driver
  app 負責檢查 driver health
```

測試簽章 driver 只保留給開發版。正式可攜產品必須使用正式簽章流程，不能要求一般使用者手動開 Windows Test Mode。

| 變更 | 行為 |

## 18. 測試計畫

### 18.1 新增單元測試

```text
tests/test_audio_frame_format_adapter.py

```

### 18.2 核心測試項目

#### format adapter

- stereo 48k → ASR mono 16k
- mono 48k → ASR mono 16k
- stereo 44.1k → output stereo 48k
- mono 16k → output stereo 48k
- limiter prevents clipping

#### direct passthrough

- stereo input remains stereo
- mono input duplicates to stereo
- ASR manager not called
- translator not called
- TTS queue not called
- no transcript created
- passthrough gain applies
- TTS synth failure does not affect passthrough

#### meeting monitor

- driver not initialized
- bridge not initialized
- virtual sink not initialized
- input source starts
- loopback source starts
- ASR event writes to meeting_original
- translation writes to meeting_translated
- TTS voice none still translates

#### dialogue voice

- virtual speaker source starts
- virtual microphone sink starts
- remote_to_local translated_tts calls ASR/LLM/TTS
- remote_to_local direct_passthrough skips ASR/LLM/TTS
- local_to_remote direct_passthrough writes to virtual mic
- local_to_remote translated_tts writes TTS to virtual mic

#### config migration

- old `auto` → explicit language
- old flat runtime fields → new meeting/dialogue config
- missing session mode → meeting_monitor
- old tts voice none → dialogue output_policy direct_passthrough
- meeting mode voice none → does not disable ASR/LLM

## 21. 最終產品目標

SyncTranslate 產品化後應具備以下特性：

```text
1. 無 driver 也能當會議監聽字幕翻譯工具。
2. 有 driver 時能進入雙向語音口譯模式。
```
