## 短期優化（無需更改架構，調整設定與參數）

### ✅ S-1：調高 `final_history_seconds`（降低長句截斷頻率）

> **已完成**：`final_history_seconds` 已更新為 `12`（`config.example.yaml` 與 `schema.py` 預設值）。

**問題**  
`final_history_seconds = 6`（預設）使得超過 6 秒的句子只能辨識末尾 6 秒音訊，前段文字消失。雖然 `_recover_final_prefix` 已補救部分情形，但根本解法是加大窗口。

**修改位置**  
`app/infra/config/schema.py`，`AsrStreamingSettings` dataclass

```python
# 修改後（已完成）
final_history_seconds: int = 12
```

同步修改 `config.example.yaml` 兩處（local 與 remote）：
```yaml
# 修改後（已完成）
final_history_seconds: 12
```

**驗收條件**  
- 說一句超過 10 秒的長句，final 文字應完整，不截頭  
- `worker_v2.py` 的 `_limited_audio` 上限 `self._final_history_seconds` 值為 12

### ⬜ S-2：為 final 啟用 `condition_on_previous_text`

**問題**  
`final_condition_on_previous_text = False`（預設），Whisper 每句獨立辨識，無法利用前句語境修正術語拼字與連貫性。

**修改位置**  
`app/infra/config/schema.py`，`AsrConfig` dataclass

```python
# 修改前
final_condition_on_previous_text: bool = False

# 修改後
final_condition_on_previous_text: bool = True
```

同步修改 `config.example.yaml` 兩處（local 與 remote）：
```yaml
# 修改前
final_condition_on_previous_text: false

# 修改後
final_condition_on_previous_text: true
```

**注意**  
`condition_on_previous_text=True` 在某些罕見情境下可能造成 Whisper 重複上一句文字（幻覺），可搭配 M-1 的 `initial_prompt` 機制降低此風險。若使用中出現重複句，可改為只在中文 final 開啟，英文保持 False。

**驗收條件**  
- 連說兩句含相同術語，第二句辨識結果應更準確  
- `faster_whisper_adapter.py` 的 `transcribe_final_result` 中 `condition_on_previous_text` 為 True

### ⬜ M-2：Whisper 辨識後文字傳回 `initial_prompt` 供下一句使用

**問題**  
即使啟用 `condition_on_previous_text=True`，Whisper 的「前文記憶」只在同一個 `transcribe()` 呼叫的 segments 之間有效，跨 final 事件無法共享。需要在應用層維護一個滑動文字緩衝，手動傳入 `initial_prompt`。

**修改位置**

#### 1. `app/infra/asr/faster_whisper_adapter.py`
新增 `push_context(text)` 與 `_context_prompt` 管理：
```python
# 在 FasterWhisperEngine dataclass 新增（non-init field）
_context_window: list[str] = field(default_factory=list, init=False, repr=False)
_context_max_chars: int = field(default=200, init=False, repr=False)

def push_context(self, text: str) -> None:
    """收到 final 文字後呼叫，更新滑動上下文視窗。"""
    stripped = text.strip()
    if not stripped:
        return
    self._context_window.append(stripped)
    # 維持總字元數 <= _context_max_chars
    while sum(len(t) for t in self._context_window) > self._context_max_chars and len(self._context_window) > 1:
        self._context_window.pop(0)

def _effective_initial_prompt(self) -> str:
    """合併 config 靜態 prompt 與動態上下文視窗。"""
    parts = []
    if self.initial_prompt.strip():
        parts.append(self.initial_prompt.strip())
    if self._context_window:
        parts.append(" ".join(self._context_window))
    return " ".join(parts)
```

在 `_transcribe` 的 kwargs 組裝改為：
```python
effective_prompt = self._effective_initial_prompt()
if effective_prompt:
    kwargs["initial_prompt"] = effective_prompt
```

#### 2. `app/infra/asr/backend_v2.py` — `FasterWhisperStreamingBackend.transcribe_final`
final 辨識完成後呼叫 `engine.push_context`：
```python
def transcribe_final(self, audio, sample_rate, *, frontend_stats=None):
    result = self._engine.transcribe_final_result(audio, sample_rate)
    text = result.text
    # ... post_processor 邏輯不變 ...
    # 更新語境視窗供下一句使用
    if text.strip():
        self._engine.push_context(text.strip())
    return BackendTranscript(...)
```

**驗收條件**  
- 連說兩句相關內容，第二句中出現的術語辨識正確率提升  
- `push_context` 不影響 partial 辨識（partial 不呼叫 push_context）

### ⬜ M-3：字元速率（CPS）估算，作為語速自適應補充

**問題**  
現有 `_recompute_adaptive_tuning` 只看音訊長度（ms），不知道辨識出幾個字元，無法估算說話者的語速（chars/sec）。快語速的說話者詞間停頓約 80~120ms，與慢語速者的「句間思考停頓」長度相近，容易誤切。

**修改位置**  
`app/infra/asr/stream_worker.py`（`StreamingAsr` class）

#### 1. 新增字元速率追蹤狀態
在 `__init__` 加入：
```python
self._adaptive_cps_samples: deque[float] = deque(maxlen=8)  # chars per second
```

#### 2. 在 `_emit_final` 計算 CPS
在 final 事件發出後，計算 CPS 並加入樣本：
```python
# 在 _emit_final 的 on_event 呼叫之後：
if text and audio_ms > 0:
    cps = len(text) / (audio_ms / 1000.0)
    self._adaptive_cps_samples.append(cps)
    self._record_final_adaptation(audio_ms=audio_ms, latency_ms=latency_ms, now_ms=now_ms)
```

#### 3. 在 `_recompute_adaptive_tuning` 使用 CPS
```python
avg_cps = (
    sum(self._adaptive_cps_samples) / len(self._adaptive_cps_samples)
    if self._adaptive_cps_samples else 0.0
)

# 快語速（中文 > 8 字/秒，英文 > 15 字/秒）：靜音閾值再收緊
if avg_cps > 8.0 and "short_turn" in mode:
    min_silence_duration_ms = max(160, int(min_silence_duration_ms * 0.85))
    mode_parts.append("fast_speaker")
```

**驗收條件**  
- 讓語速明顯較快的說話者連說幾句後，`adaptive_mode` 應出現 `short_turn+fast_speaker`  
- 語速正常的說話者不觸發此模式

## 長期優化（架構層改動）

### ✅ L-1：換用 `whisper-large-v3-turbo`（最直接的延遲改善）

> **已完成**：`config.example.yaml` remote 通道已設定 `model: large-v3-turbo`。local 通道使用 FunASR，不適用此項。


**問題**  
`large-v3` 的 decoder 有 32 個 transformer blocks，即使有 GPU 加速，推論時間仍是 partial 延遲的主要來源。`large-v3-turbo` 只有 4 個 decoder blocks，精度損失 < 1% WER，但速度快 **6~8 倍**。

**修改位置**

#### 1. `config.example.yaml`
```yaml
asr_channels:
  local:
    model: large-v3-turbo   # 修改前: large-v3
  remote:
    model: large-v3-turbo   # 修改前: large-v3
```

#### 2. `app/ui/pages/local_ai_page.py` — model combo box 選項
找到 model combo box 初始化區段，加入 `large-v3-turbo` 選項：
```python
# 在 asr_model_combo 與 remote_asr_model_combo 的 addItems 清單中加入
"large-v3-turbo"
```

#### 3. `app/infra/asr/faster_whisper_adapter.py` — model 欄位預設值
```python
# 修改前
model: str = "large-v3"

# 修改後
model: str = "large-v3-turbo"
```

**驗收條件**  
- 使用 `large-v3-turbo`，partial 出現延遲應 < 150ms（GPU，float16）  
- 英文 WER 與 `large-v3` 差距 < 2%（可用現有 benchmark 工具驗證）  
- UI 可正常選擇並切換此模型

### ⬜ L-2：FunASR SenseVoice Streaming 模式（中文 CTC 真正串流）

**問題**  
FunASR 的 `iic/SenseVoiceSmall` 目前以批次模式呼叫（每 15s chunk 才跑一次），不是真正的串流。FunASR 支援 `online` 模式，可以做到逐幀 CTC 解碼，延遲可降至 200~400ms。

**修改位置**

#### 1. `app/infra/asr/backend_v2.py` — `FunASRStreamingBackend`
新增 `_online_mode: bool` 旗標與對應的 `transcribe_partial` 分支：
```python
class FunASRStreamingBackend:
    def __init__(self, ..., online_mode: bool = False):
        ...
        self._online_mode = online_mode
        self._online_cache = None  # FunASR online inference cache

    def transcribe_partial(self, audio, sample_rate, *, frontend_stats=None):
        if self._online_mode:
            return self._transcribe_online(audio, sample_rate, is_final=False)
        # 現有批次邏輯不變
        ...

    def _transcribe_online(self, audio, sample_rate, *, is_final: bool) -> BackendTranscript:
        """使用 FunASR AutoModel 的 online/cache 模式進行串流辨識。"""
        model = self._get_model()
        # FunASR online API: model.generate(input=audio, cache=self._online_cache, is_final=is_final)
        result = model.generate(
            input=audio,
            cache=self._online_cache,
            language="zh",
            use_itn=True,
            batch_size_s=60,
            is_final=is_final,
        )
        self._online_cache = result.get("cache", self._online_cache)
        text = _extract_funasr_text(result)
        return BackendTranscript(text=text, is_final=is_final, detected_language="zh")
```

#### 2. `app/infra/config/schema.py` — 新增 `funasr_online_mode` 旗標
```python
@dataclass(slots=True)
class AsrConfig:
    ...
    funasr_online_mode: bool = False  # 啟用 FunASR streaming online 模式（實驗性）
```

#### 3. `app/infra/asr/backend_v2.py` — `_build_funasr_backend_pair`
傳入 `online_mode`：
```python
def _build_funasr_backend_pair(profile, *, language, post_processor=None):
    online = getattr(profile, "funasr_online_mode", False)
    partial_backend = FunASRStreamingBackend(..., online_mode=online)
    final_backend = FunASRStreamingBackend(..., online_mode=False)  # final 仍用批次確保準確度
    ...
```

**驗收條件**  
- 中文 partial 延遲 < 300ms（GPU）  
- 長句不再截斷（online 模式天生無 history_seconds 限制）  
- `funasr_online_mode: false` 時行為與修改前完全相同（向後相容）

### ⬜ L-3：Speculative Decoding（Whisper 草稿加速）

**問題**  
faster-whisper >= 1.0 支援 speculative decoding：用小模型（如 `distil-whisper/distil-large-v3`）出草稿，大模型驗證，吞吐量提升 **3~4x** 而精度幾乎不變。

**修改位置**

#### 1. `app/infra/config/schema.py` — 新增 speculative 設定
```python
@dataclass(slots=True)
class AsrConfig:
    ...
    speculative_draft_model: str = ""   # 空字串=停用，如 "distil-large-v3"
    speculative_num_beams: int = 1
```

#### 2. `app/infra/asr/faster_whisper_adapter.py` — `FasterWhisperEngine`
新增欄位：
```python
speculative_draft_model: str = ""
speculative_num_beams: int = 1
_draft_model: object | None = field(default=None, init=False, repr=False)
```

在 `_get_model` 後若 `speculative_draft_model` 非空，初始化 draft model：
```python
def _get_draft_model(self):
    if not self.speculative_draft_model.strip():
        return None
    # 使用與主模型相同的 device/compute_type
    from faster_whisper import WhisperModel
    return WhisperModel(
        self.speculative_draft_model,
        device=self._runtime_device or self.device,
        compute_type=self._runtime_compute_type or self.compute_type,
    )
```

在 `_transcribe` 的 `model.transcribe` 呼叫前，若有 draft model 則加入：
```python
draft = self._get_draft_model()
if draft is not None:
    kwargs["assistant_model"] = draft
```

#### 3. `app/infra/asr/backend_v2.py` — `_build_engine` 傳遞
```python
def _build_engine(profile: AsrConfig, *, language: str) -> FasterWhisperEngine:
    return FasterWhisperEngine(
        ...  # 現有欄位不變
        speculative_draft_model=getattr(profile, "speculative_draft_model", ""),
        speculative_num_beams=getattr(profile, "speculative_num_beams", 1),
    )
```

#### 4. `config.example.yaml` — 範例設定
```yaml
asr_channels:
  remote:
    # Speculative Decoding（需下載 draft model，GPU 記憶體需求增加約 1.5GB）
    speculative_draft_model: ""   # 如填入 "distil-large-v3" 可加速 3~4x
```

**驗收條件**  
- 設定 `speculative_draft_model: distil-large-v3` 後，final 推論時間應縮短 50% 以上  
- 未設定時不影響現有功能  
- `RuntimeError` 時（如 draft model 不存在）應 fallback 至無 speculative 模式，並 log warning
