# 音訊裝置建議配置

## 重要安全狀態

自製 SyncTranslate Windows 音訊 driver 目前只允許在一次性 Windows VM / lab 環境驗證，不建議安裝到日常使用主機。已觀察到 `tabletaudiosample.sys` 觸發 `SYSTEM_THREAD_EXCEPTION_NOT_HANDLED (0x7E)` 藍畫面；因此 MSI 已改為只封裝檔案，不會自動安裝 driver 或排定重開機，安裝腳本也預設阻擋主機安裝，除非明確傳入 `-AllowHostInstall`。

# 音訊裝置建議配置

本文對齊目前的 `AudioRouteConfig`。SyncTranslate 的通話音訊路由只有一條產品主線：

| 模式 | 用途 |
| --- | --- |
| `synctranslate_virtual_audio` | 使用 `SyncTranslate Virtual Speaker` / `SyncTranslate Virtual Microphone`，不依賴第三方虛擬音訊軟體。 |

## 建議主線

自用/內測建議配置：

```yaml
audio:
  routing_mode: synctranslate_virtual_audio
  virtual_audio:
    speaker_name: SyncTranslate Virtual Speaker
    microphone_name: SyncTranslate Virtual Microphone
    bridge_enabled: true
    bridge_path: runtimes\audio\sync_audio_bridge.exe
    sample_rate: 48000
    frame_ms: 10
    target_latency_ms: 120
    require_driver: true
  system_devices:
    capture_role: communications
    render_role: communications
    follow_system_default: true
    fallback_to_multimedia_role: true
    exclude_virtual_devices: true
  call_translation:
    listen_remote_original: true
    listen_remote_translation: true
    output_local_original: false
    output_local_translation: true
```

通訊軟體中選：

```text
Speaker: SyncTranslate Virtual Speaker
Microphone: SyncTranslate Virtual Microphone
```

注意：Windows 可能會依語系把 speaker endpoint 顯示成 `喇叭 (SyncTranslate Virtual Audio Device)`。只要名稱含有 SyncTranslate，且它是輸出/喇叭端點，即可作為 `SyncTranslate Virtual Speaker` 使用。

SyncTranslate App 內部會把：

- 遠端音訊從 `SyncTranslate Virtual Speaker` 讀入。
- 本地麥克風從 Windows 預設通訊麥克風讀入。
- 對方翻譯與對方原聲播放到真實耳機。
- 我的翻譯寫入 `SyncTranslate Virtual Microphone` 給會議軟體。

目前 App、Bridge、Driver 與 PCM 閉環都已落地。免費自用版採 test-signed driver，重點是確認 Test Mode 安裝、release 打包與自己常用通話軟體可用。

## 是否需要額外安裝

需要。Windows 通訊軟體要看見新的：

```text
SyncTranslate Virtual Speaker
SyncTranslate Virtual Microphone
```

必須先安裝 SyncTranslate 的 Windows 音訊 driver。單純啟動 Python App 或 `main.py` 不能在系統中建立新的可選麥克風/喇叭端點。

自用版流程：

1. 下載或建置 App zip。
2. 以 Administrator 權限安裝 test-signed `SyncTranslate Audio Driver`。
3. 依需要開啟 Windows Test Mode；Secure Boot 可能需要關閉。
4. 啟動 SyncTranslate App，App 會自動啟動 bridge。
5. 通訊軟體選 `SyncTranslate Virtual Speaker` 與 `SyncTranslate Virtual Microphone`。

若沒有安裝 driver，`synctranslate_virtual_audio` 只能顯示設定與診斷，不能完成 full-duplex 音訊路由。此時請安裝或修復 SyncTranslate Audio Driver，App 不會回退到第三方虛擬音訊工具。

## Bridge

App 端 client 會優先啟動 `virtual_audio.bridge_path` 指向的 `sync_audio_bridge.exe`，並透過 Windows Named Pipe 溝通；若 exe 尚不存在，開發環境可 fallback 啟動 Python worker：

```text
python -m app.infra.audio.bridge_process --pipe \\.\pipe\SyncTranslateAudioBridge
```

若要手動 smoke test，也可以使用 stdin/stdout debug 模式：

```powershell
uv run python -m app.infra.audio.bridge_process --stdio
```

若要先產出獨立 worker，請使用 uv 驅動的打包腳本：

```powershell
tools/runtime_setup/package_audio_bridge.ps1
```

腳本內部執行的是：

```powershell
uv run --group build pyinstaller --onefile --clean --name sync_audio_bridge app/infra/audio/bridge_process.py
```

輸出位置是：

```text
runtimes/audio/sync_audio_bridge.exe
```

Bridge 目前負責 named pipe 控制面、shared memory PCM ring buffer、Windows named event、stats 與 virtual microphone write path。Driver 端已透過 shared memory / event 合約接上 PCM 閉環；自用版需確認 MSI 內的 test-signed driver 與 release build 中的 bridge 版本一致。

## Driver 安裝方式

目前已可產出 test-signed driver package、PowerShell 安裝腳本，以及 MSI wrapper。MSI 不會由 App 靜默安裝，仍由你明確執行。

原因是 Windows 音效 driver 安裝會改動系統 driver store、需要 Administrator 權限，測試簽章還需要 Windows Test Mode。開發階段保留明確的安裝腳本，同時提供 MSI 作為較接近最終使用者體驗的包裝。

Driver 工作目錄：

```text
drivers/synctranslate_virtual_audio
```

檢查本機是否具備 WDK/簽章工具：

```powershell
drivers/synctranslate_virtual_audio/scripts/check_prereqs.ps1
```

WDK 準備好後的建置流程：

```powershell
drivers/synctranslate_virtual_audio/scripts/build_driver_msi.ps1
```

若要拆開執行，也可以依序跑 `fetch_sysvad.ps1`、`build_driver_poc.ps1`、`sign_driver_package.ps1`、`validate_driver_package.ps1`、`package_driver_msi.ps1`。目前開發建置預設關閉 `SpectreMitigation`，避免缺少 Spectre libraries 時建置失敗。

安裝需在 Administrator PowerShell 執行：

```powershell
Set-Location E:\py\SyncTranslate
.\drivers\synctranslate_virtual_audio\scripts\enable_test_mode.ps1
# reboot
.\drivers\synctranslate_virtual_audio\scripts\install_driver_package.ps1
```

若已產出 MSI，也可以改由 Administrator 權限直接執行：

```text
artifacts/driver/synctranslate_virtual_audio/SyncTranslateVirtualAudioDriver.msi
```

目前這個 MSI 是 test-signed 自用/內測 installer，不需要付費憑證；代價是使用者必須接受 Test Mode 與可能的 SmartScreen/防毒提醒。

若 `enable_test_mode.ps1` 顯示「值受安全開機原則保護，因此無法修改或刪除」，代表 Secure Boot 阻擋 Test Mode。自用/內測需先進 BIOS/UEFI 關閉 Secure Boot；若有啟用 BitLocker，請先暫停 BitLocker 或確認已保存修復金鑰。

## 通話翻譯開關

| 設定 key | 意義 | 預設 |
| --- | --- | --- |
| `listen_remote_original` | 我聽到對方原聲 | true |
| `listen_remote_translation` | 我聽到對方翻譯 | true |
| `output_local_original` | 對方聽到我的原聲 | false |
| `output_local_translation` | 對方聽到我的翻譯 | true |

若同時開啟 `output_local_original` 與 `output_local_translation`，對方會聽到原聲與翻譯兩路聲音，容易覺得重複。

## 四路欄位

| 設定 key | 用途 |
| --- | --- |
| `audio.meeting_in` | 遠端/會議音訊輸入 |
| `audio.microphone_in` | 本地音訊輸入 |
| `audio.speaker_out` | 本地喇叭輸出 |
| `audio.meeting_out` | 送回會議端的輸出 |

路由級音量 key 已移除。SyncTranslate 不再調整四路音量，請直接使用 Windows 系統音量與通訊軟體音量控制。

這些欄位可由 App 自動填入，用於診斷與內部路由，不要求使用者手動選四個裝置。

## 避免回音

- 使用耳機比喇叭穩定。
- `synctranslate_virtual_audio` 預設不把我的翻譯播給自己，只寫到 Virtual Microphone。
- 通訊軟體請選 SyncTranslate virtual speaker / microphone，不要選實體麥克風或耳機作為會議端點。

相關 runtime key：

- `local_echo_guard_enabled`
- `local_echo_guard_resume_delay_ms`
- `remote_echo_guard_resume_delay_ms`

## queue overflow 排查

看到：

```text
ASR queue overflow (local): dropped 11 chunks total
```

代表 ASR 已丟音訊。排查順序：

1. 確認 `runtime.asr_queue_maxsize_local` / `remote` 是 `256`。
2. 確認 `runtime.asr_profile_local` / `remote` 是 `meeting_room`。
3. 若只需要單向翻譯，暫時把另一側 ASR 語言設為 `none`。
4. 降低 TTS 或翻譯負載。
5. 檢查 GPU 是否同時被其他程式占用。
6. 匯出 session report，查看 `stats.asr.*.dropped_chunks` 是否持續增加。
