# 音訊裝置建議配置

這份文件說明 SyncTranslate 在 Windows + VoiceMeeter 下常見的裝置配置方式。

## 基本原則

SyncTranslate 目前固定有兩條邏輯通道：

- `remote`
- `local`

它們各自有獨立的：

- 輸入來源
- ASR 語言
- 翻譯目標
- TTS 輸出

因此在規劃音訊裝置時，重點是先想清楚：

- 哪個輸入要當作遠端來源
- 哪個輸入要當作本地來源
- 哪個輸出要回送給本地耳機 / 喇叭
- 哪個輸出要回送給會議軟體

## 建議對應

### SyncTranslate 輸入

- `meeting_in`
  - 遠端來源
- `microphone_in`
  - 本地來源

### SyncTranslate 輸出

- `speaker_out`
  - 回送到本地監聽
- `meeting_out`
  - 回送到會議端

## VoiceMeeter 常見做法

### 遠端來源

會議軟體輸出 -> VoiceMeeter -> `meeting_in`

### 本地來源

麥克風 / 本地輸入 -> VoiceMeeter 或實體裝置 -> `microphone_in`

### 回送給本地

`speaker_out` -> 本地耳機 / 喇叭或對應虛擬 bus

### 回送給遠端

`meeting_out` -> VoiceMeeter Input / AUX Input -> 會議軟體麥克風

## B1 / B2 測試情境

如果你刻意把同一個音源同時送到 B1 與 B2，常見用途是：

- 比較本地與遠端兩條通道的 ASR 結果
- 驗證兩條通道是否互相干擾

這種測法本身是合理的，但要注意：

- 兩條通道使用相同音源時，逐字稿理論上應該接近
- 若兩邊結果差很多，通常要檢查：
  - `resolved_backend`
  - `device_effective`
  - `speech_started_count`
  - `soft_endpoint_count`
  - `hard_endpoint_count`
  - `endpoint_signal.pause_ms`
  - `postprocessor.final.rejected_count`
  - `dropped_chunks`

## 雙通道同時發話

真實使用情境中，本地與遠端確實可能同時發話。  
SyncTranslate 現在邏輯上已避免共享模型互相污染，但若：

- 本地中文 -> faster-whisper
- 遠端中文 -> faster-whisper
- `device_effective = cpu`

那麼即使不互相污染，仍可能因為 CPU 吞吐量不足而出現延遲上升。

## 建議驗收方式

1. 先做單通道測試
2. 再做雙通道同時發話測試
3. 匯出 `session_report`
4. 檢查是否為：
   - 路由錯誤
   - backend 不一致
   - CPU bottleneck
   - endpoint 過度敏感
   - validator 正在大量擋掉垃圾輸出
   - passthrough / playback 失敗

## 一句話建議

如果你要追求雙通道中文同時發話的體驗，最關鍵的不只是路由正確，還包括：

- 正確的 backend 路由
- 可用的 CUDA 環境
- 健康的輸出裝置
