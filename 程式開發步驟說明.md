# 程式開發步驟說明
## 依據 `實作開發計畫.md` 的實作順序

## 1. 文件目的

這份文件不是產品規劃書，而是實際寫程式時要照著走的工程順序。

目標是：

1. 不一次把所有功能混在一起做。
2. 每一步都能單獨驗證。
3. 每一步都知道該建立哪些檔案。
4. 出問題時能快速判斷是音訊路由、模型、UI，還是會議軟體設定。

---

## 2. 開發原則

這個專案最容易出錯的地方不是翻譯，而是音訊路由。

所以工程順序要遵守以下原則：

1. 先確認裝置與路由，再接 ASR。
2. 先做接收端單向流程，再做發話端單向流程。
3. 先做 CLI / debug 版，再包成完整 UI。
4. 每完成一步都要有可重現的測試方法。
5. 不要一開始就做雙向全開。

---

## 4. 先決條件

### 4.1 環境

1. Python 3.11+
2. VoiceMeeter Banana 已安裝
3. Banana-only 路由已手動測通
4. 已知各裝置名稱

### 4.2 第一版建議依賴

1. `PySide6`
2. `sounddevice`
3. `numpy`
4. `PyYAML`
5. `loguru` 或內建 `logging`

若後面要接模型，再補：

1. `faster-whisper`
2. 翻譯 API SDK
3. TTS 相關套件

---

## 5. 第一階段：建立最小骨架

### 5.1 目標

先把專案跑起來，能開視窗、能讀設定檔、能列出裝置。

### 5.2 要做的檔案

1. `pyproject.toml`
2. `main.py`
3. `app/settings.py`
4. `app/schemas.py`
5. `config.example.yaml`

### 5.3 要完成的內容

1. 建立應用程式入口。
2. 建立設定檔載入。
3. 定義邏輯通道資料結構。
4. 顯示基本主視窗。

### 5.4 建議資料結構

至少先定義：

1. `AudioRouteConfig`
2. `LanguageConfig`
3. `AppConfig`
4. `DeviceInfo`

### 5.5 完成標準

1. 能執行 `main.py`
2. 主視窗能打開
3. 能讀到 `config.example.yaml`

---

## 6. 第二階段：裝置列舉與設定頁

### 6.1 目標

能看到所有可用音訊裝置，並手動指定四個邏輯通道。

### 6.2 要做的檔案

1. `app/device_manager.py`
2. `app/pages_audio_routing.py`
3. `app/pages_quick_start.py`

### 6.3 `device_manager.py` 要做什麼

1. 列出所有輸入裝置
2. 列出所有輸出裝置
3. 顯示裝置 index、名稱、channel、預設 sample rate
4. 能依名稱搜尋 `VoiceMeeter` 相關裝置

### 6.4 UI 要做到什麼

`音訊路由` 分頁至少要有：

1. `remote_in` 下拉選單
2. `local_mic_in` 下拉選單
3. `local_tts_out` 下拉選單
4. `meeting_tts_out` 下拉選單
5. `載入 Banana 預設` 按鈕
6. `儲存設定` 按鈕

### 6.5 完成標準

1. UI 能正確列出 VoiceMeeter 相關裝置
2. 你能手動選好四個邏輯通道
3. 設定可存檔與重載

---

## 7. 第三階段：路由檢查器

### 7.1 目標

在真正開始處理音訊前，先攔住明顯錯誤配置。

### 7.2 要做的檔案

1. `app/router.py`
2. `app/debug_panel.py`

### 7.3 `router.py` 要檢查的規則

至少先做這些：

1. `remote_in` 不能等於 `meeting_tts_out`
2. `remote_in` 不能等於 `local_tts_out`
3. `local_mic_in` 不能空白
4. `meeting_tts_out` 必須是輸出裝置
5. `remote_in` 必須是輸入裝置

### 7.4 UI 呈現

1. 顯示 `OK / Warning / Error`
2. 顯示是哪個欄位衝突
3. 不通過時禁止按 `開始`

### 7.5 完成標準

1. 常見錯路由會被攔下
2. 不需要真的開始串音訊就能先發現配置錯誤

---

## 8. 第四階段：只做 remote_in 擷取

### 8.1 目標

先證明程式能穩定抓到會議軟體的對方聲音。

### 8.2 要做的檔案

1. `app/audio_capture.py`
2. `app/pages_diagnostics.py`

### 8.3 功能要求

1. 指定 input device 後開始讀音訊
2. 顯示即時音量條
3. 顯示目前 sample rate
4. 顯示 frame 計數

### 8.4 測試方法

1. 會議軟體 Speaker 設成 `VoiceMeeter Input`
2. `remote_in` 設成 `VoiceMeeter Aux Output`
3. 讓會議中有人說話或播放測試音
4. 看 UI 的 remote 電平是否跳動

### 8.5 完成標準

1. 程式能穩定抓到 `remote_in`
2. 停止與重啟 capture 不會卡死

---

## 9. 第五階段：只做 local_mic_in 擷取

### 9.1 目標

把你的實體麥克風獨立接進來。

### 9.2 要做的檔案

1. `app/audio_capture.py`
2. `app/pages_diagnostics.py`

### 9.3 功能要求

1. 顯示 local mic 即時電平
2. 能切換不同麥克風
3. 能靜音
4. 能在設備失效後重新綁定

### 9.4 完成標準

1. `local_mic_in` 穩定可用
2. 不影響 `remote_in` 捕捉

---

## 10. 第六階段：先做 UI 主控台

### 10.1 目標

在還沒接 ASR 之前，先把 UI 骨架做好。

### 10.2 要做的檔案

1. `app/ui_main.py`
2. `app/pages_quick_start.py`
3. `app/pages_audio_routing.py`
4. `app/pages_live_caption.py`
5. `app/pages_models.py`
6. `app/pages_diagnostics.py`

### 10.3 分頁要求

#### 快速開始

1. 顯示 Banana-only 推薦設定
2. 一鍵套用推薦裝置
3. 顯示會議軟體應如何設定 Speaker / Mic

#### 音訊路由

1. 四個邏輯通道下拉選單
2. 每個通道的測試按鈕
3. 電平條

#### 即時字幕

1. Remote Original
2. Remote Translated
3. Local Original
4. Local Translated

#### 模型與語言

1. 語言選擇
2. ASR / Translate / TTS 選擇
3. 快譯 / 穩譯模式

#### 除錯與診斷

1. Banana 裝置狀態
2. 最近錯誤
3. Route check 結果

### 10.4 完成標準

1. 不看 console 也能操作
2. 音訊裝置設定、保存、重載全部可用

---

## 11. 第七階段：remote pipeline 接 ASR

### 11.1 目標

完成：

`remote_in -> 英文辨識文字`

### 11.2 要做的檔案

1. `app/asr_worker.py`
2. `app/pipeline_remote.py`
3. `app/transcript_buffer.py`

### 11.3 最小功能

1. 接收音訊 chunk
2. 呼叫英文 ASR
3. 回傳文字
4. 顯示到 `Remote Original`

### 11.4 第一版建議

1. 先不用 partial
2. 先做短段送出
3. 先追求穩定，不追求超低延遲

### 11.5 完成標準

1. 對方說英文時，畫面會出現英文辨識字
2. 錯誤時不會讓整個 UI 當掉

---

## 12. 第八階段：remote pipeline 接翻譯

### 12.1 目標

完成：

`remote_in -> 英文辨識 -> 中文翻譯文字`

### 12.2 要做的檔案

1. `app/translate_worker.py`
2. `app/pipeline_remote.py`

### 12.3 功能要求

1. 將 remote 英文文字送翻譯器
2. 產生中文翻譯
3. 顯示到 `Remote Translated`

### 12.4 完成標準

1. 畫面同時看到英文原文與中文翻譯
2. 原文與譯文對應順序正確

## 13. 第九階段：remote pipeline 接中文 TTS

### 13.1 目標

`remote_in -> 英文 ASR -> 中文翻譯 -> 中文 TTS -> local_tts_out`

### 13.2 要做的檔案

2. `app/audio_playback.py`
3. `app/pipeline_remote.py`

### 13.3 功能要求

1. 中文文字轉語音

### 13.4 完成標準

1. 你能在耳機中聽到中文譯音
2. 這段中文不會進會議麥克風

## 15. 第十一階段：local pipeline 接英文 TTS 與 meeting_tts_out

### 15.1 目標

### 15.5 完成標準

2. 這段英文不會回到 `remote_in`

## 19. 每一步完成後都要做的驗證

### 19.1 裝置層驗證

1. Banana 裝置名稱是否仍存在
2. `remote_in` 是否仍能收到音
3. `meeting_tts_out` 是否仍能送出音

### 19.2 UI 層驗證

1. 設定是否能保存
2. 重開程式是否能還原
3. 畫面是否顯示正確欄位

### 19.3 pipeline 驗證

1. remote pipeline 是否獨立可跑
2. local pipeline 是否獨立可跑
3. 同時跑時是否衝突

---

## 20. 最建議的實際開發順序

如果你現在就開始寫，建議順序如下：

1. 建專案骨架與設定檔
2. 做裝置列舉
3. 做路由檢查器
4. 做 remote_in 電平監看
5. 做 local_mic_in 電平監看
6. 做主 UI 分頁骨架
7. 接 remote ASR
8. 接 remote 翻譯
9. 接 remote 中文 TTS
10. 接 local ASR
11. 接 local 翻譯
12. 接 local 英文 TTS
13. 做雙向模式整合
14. 做 partial/final 與歷史緩衝
15. 做 Banana-only 快速設定與診斷

不要把順序反過來。

尤其不要一開始就：

1. 先接所有模型
2. 先做雙向同開
3. 先做 fancy UI

這樣最容易卡在路由問題卻找不到原因。

## 21. 第一版完成標準

這份開發步驟全部跑完後，第一版程式至少應具備：

1. 可設定四個邏輯通道
2. 可列出 Banana 相關裝置
3. 可抓到 `remote_in`
4. 可抓到 `local_mic_in`
5. 可顯示四欄字幕
6. 可播放中文譯音到耳機
7. 可播放英文譯音到會議麥克風
8. 可在單向與雙向模式之間切換
9. 可做基本路由診斷
