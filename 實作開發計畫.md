## 1. 這版計畫的核心修正

這份文件已改成「個人自用版」規劃，與前一版有幾個根本差異：

1. 不再以商業產品為目標。
2. 不再以 Google Meet、Teams、Zoom 的平台整合為主軸。
3. 不再優先處理法規、商業模式、企業管理、成本控管、多人租戶等內容。
4. 研發重心改成「只要會議軟體能選音訊裝置，就能接入」。
5. 系統形態從「Web + Extension + Backend」改成「Windows 本地桌面程式 + 虛擬音訊路由」。

這樣的好處很明確：

1. 技術路徑更直接。
2. 不會被平台 API 權限卡住。
3. 一次做對音訊路由後，理論上可支援多數會議軟體。
4. 開發時可先在本機完全測通，不必先處理平台審核或帳號權限。

## 2. 可行性判斷

### 2.1 結論

可行，而且比平台整合更務實。

你提出的方向本質上是：

1. 把會議軟體當成黑盒子。
2. 不管它是 Meet、Teams、Zoom、Discord、Skype、Webex 或任何桌面程式。
3. 只要它能設定 Speaker 與 Microphone 裝置，就透過 Windows 音訊裝置把訊號導到你的翻譯程式。

這個方向是正確的。

### 2.2 需要修正的一點

你原本描述中：

`本地語音(中文)→B1(麥克風)→AI(語音辨識+翻譯)→A1(視訊會議音訊輸出)→對方聽到英文`

這一段的最後目標不應該是 `A1`。

原因是：

1. `A1/A2/A3` 在 VoiceMeeter 裡是硬體輸出。
2. 視訊會議軟體要吃的是「錄音裝置 / 虛擬麥克風」。
3. 所以送給對方的英文 TTS，應該送進 `B1` 或另一條獨立的虛擬麥克風裝置。

也就是說，更正後應該是：

`本地語音(中文) -> AI(辨識+翻譯+英文TTS) -> B1 或獨立虛擬麥克風 -> 視訊會議軟體 -> 對方聽到英文`

### 2.3 真正的技術難點

這個方向可行，但真正要解的是以下四件事：

1. 避免回授與迴音。
2. 將「對方聲音」與「我要送給對方的翻譯聲音」完全隔離。
3. 做出雙向同時運作時的低延遲。
4. 讓字幕、原文辨識與譯後語音之間保持一致。

## 3. 建議的總體方案

### 3.1 總體原則

本版建議採用以下原則：

1. 以 Windows 音訊裝置整合為核心，不依賴特定會議平台。
2. 翻譯程式採本地桌面程式型態。
3. 所有音訊路由先用「邏輯通道名稱」設計，再映射到 VoiceMeeter 裝置。
4. 先做單向收聽翻譯，再做單向發話翻譯，最後才做全雙工。
5. 先把音訊路由測通，再追求模型品質與介面美化。

### 3.2 邏輯通道設計

不要在程式內直接把 A1/A2/B1/B2 寫死成業務邏輯。建議先用以下邏輯名稱：

| 邏輯通道 | 說明 |
| --- | --- |
| `remote_in` | 對方講話的音訊來源 |
| `local_mic_in` | 你自己的麥克風來源 |
| `local_tts_out` | 你自己要聽的中文譯後語音輸出 |
| `meeting_tts_out` | 翻譯程式送出的英文 TTS 輸出，最終會被會議軟體當成麥克風來源 |

這版不需要 `raw_remote_monitor`。也就是說，對方原音不需要同時播到你耳機；你只看英文辨識字與中文翻譯，並只聽中文譯音。

程式內只處理這些邏輯通道，真正對應到哪個 VoiceMeeter 裝置或 bus，再由設定檔映射。

### 3.3 為什麼這比平台整合更好

1. 不需要每個平台各寫一套 Adapter。
2. 不怕平台 API 頻繁改版。
3. 不怕會議平台限制擷取其他人的媒體流。
4. 可同時支援桌面 App 與瀏覽器版會議。
5. 更容易在本機除錯。

## 4. 最推薦的音訊拓樸

### 4.1 結論

你現在的需求其實可以先不用額外硬體，也不一定要先裝額外 VB-CABLE。

這裡的「兩條獨立虛擬音訊線」不是實體線，不需要把電腦喇叭真的接兩條線出去。它指的是 Windows 裡的兩組虛擬音訊端點，用軟體方式把聲音從一個程式送到另一個程式。

對你目前的需求，最建議先採用：

`VoiceMeeter Banana 內建兩組虛擬 I/O + 實體耳機`

也就是：

1. 第一條虛擬路徑負責「會議軟體 Speaker -> 翻譯程式 remote_in」。
2. 第二條虛擬路徑負責「翻譯程式英文 TTS -> 會議軟體 Microphone」。
3. 中文譯音直接輸出到你的實體耳機，不混回會議。

這是目前最乾淨、最省事、最符合你需求的版本。

### 4.2 VoiceMeeter Banana 是否足夠

可以，而且對你這版需求是足夠的。

根據 VoiceMeeter Banana 的官方功能說明，它本身有：

1. 3 個 Hardware Inputs。
2. 2 個 Software Inputs。
3. 3 個 Hardware Outputs。
4. 2 個 Software Outputs。

對你的需求來說，剛好可以這樣用：

1. `VoiceMeeter Input (VAIO)` 當會議軟體的 Speaker 入口。
2. `VoiceMeeter Aux Output (B2)` 當翻譯程式的 `remote_in`。
3. `VoiceMeeter AUX Input` 當翻譯程式英文 TTS 的輸出入口。
4. `VoiceMeeter Output (B1)` 當會議軟體的 Microphone。
5. 實體耳機由 A1 輸出，專門播放中文譯音。

也就是說，在你不需要同時監聽對方原音的前提下，`VoiceMeeter Banana 單獨就能成立`。

### 4.3 Banana-only 最小可行拓樸

這是我最推薦你先做的配置，因為不需要再多裝一堆東西。

#### 裝置角色

| 用途 | 建議裝置 |
| --- | --- |
| 會議軟體 Speaker | `VoiceMeeter Input (VB-Audio VoiceMeeter VAIO)` |
| 翻譯程式 `remote_in` | `VoiceMeeter Aux Output (VB-Audio VoiceMeeter AUX VAIO)` |
| 會議軟體 Microphone | `VoiceMeeter Output (VB-Audio VoiceMeeter VAIO)` |
| 翻譯程式 `meeting_tts_out` | `VoiceMeeter AUX Input (VB-Audio VoiceMeeter AUX VAIO)` |
| 翻譯程式 `local_mic_in` | 你的實體麥克風 |
| 翻譯程式 `local_tts_out` | 你的實體耳機或耳麥耳機輸出 |
| A1 | 你的實體耳機 |

#### 對方講話路徑

`會議軟體 Speaker -> VoiceMeeter Input (VAIO) -> B2 -> 翻譯程式 remote_in -> 英文 ASR -> 顯示英文 + 中文對照 -> 中文 TTS -> 實體耳機`

#### 你講話路徑

`實體麥克風 -> 翻譯程式 local_mic_in -> 中文 ASR -> 英文翻譯 -> 英文 TTS -> VoiceMeeter AUX Input -> B1 -> 會議軟體 Microphone`

#### Banana 內部路由原則

1. `VoiceMeeter Input (VAIO)` 這條 virtual strip 只開 `B2`，不要開 `A1/A2/A3/B1`。
2. `VoiceMeeter AUX Input` 這條 virtual strip 只開 `B1`，不要開 `A1/A2/A3/B2`。
3. 中文 TTS 直接由翻譯程式播放到你的耳機，不經 VoiceMeeter 虛擬輸入。
4. 實體麥克風建議由翻譯程式直接抓，不先進 VoiceMeeter 混音。

這樣的好處是：

1. 對方原音只會進翻譯，不會直接播給你。
2. 你送出的英文 TTS 只會進會議，不會回到辨識端。
3. Banana 的兩條軟體虛擬 I/O 剛好各用一條，夠用。

### 4.4 你現在這套 Banana 介面要怎麼設

根據你貼的畫面，VoiceMeeter Banana 已經能用，只是路由策略需要調整。

建議你這樣設：

#### A. Hardware Out

1. `A1` 設成你的實體耳機。
2. 若你現在用的是喇叭，建議改耳機或耳麥，避免中文 TTS 從喇叭回收到麥克風。
3. `A2/A3` 先留空，除非你真的有第二個監聽設備。

#### B. Hardware Input 1

1. 不建議把你的實體麥克風先混到 Banana 再給程式。
2. 這個 strip 可先留給人工監聽或暫時不用。
3. 如果保留，先把 `A1/A2/A3/B1/B2` 全關掉，避免誤送。

#### C. Virtual Input: VoiceMeeter Input

1. 把這條當作「會議軟體 Speaker 進來的通道」。
2. 在這條 strip 上只打開 `B2`。
3. `A1/A2/A3/B1` 全部關掉。

#### D. Virtual Input: VoiceMeeter AUX Input

1. 把這條當作「翻譯程式英文 TTS 要送回會議」的通道。
2. 在這條 strip 上只打開 `B1`。
3. `A1/A2/A3/B2` 全部關掉。

#### E. 會議軟體裝置設定

1. Speaker 設為 `VoiceMeeter Input (VB-Audio VoiceMeeter VAIO)`。
2. Microphone 設為 `VoiceMeeter Output (VB-Audio VoiceMeeter VAIO)`。

#### F. 翻譯程式裝置設定

1. `remote_in` 設為 `VoiceMeeter Aux Output (VB-Audio VoiceMeeter AUX VAIO)`。
2. `meeting_tts_out` 設為 `VoiceMeeter AUX Input (VB-Audio VoiceMeeter AUX VAIO)`。
3. `local_mic_in` 設為你的實體麥克風。
4. `local_tts_out` 設為你的實體耳機。

### 4.5 什麼情況才需要額外 VB-CABLE

如果未來你想要以下任一項，再加裝額外 VB-CABLE 會比較合理：

1. 中文 TTS 也想先進 VoiceMeeter，再做 ducking 或混音。
2. 想額外保留原會議音預覽通道。
3. 想把系統聲、會議聲、翻譯聲做更細的隔離。
4. 想讓翻譯程式完全不碰實體耳機，所有輸出都先進虛擬裝置。

若只是你現在的目標，`Banana 本身就夠，不需要額外硬體，也不需要實體分接線`。

### 4.6 如果真的要做「兩條額外虛擬線」，具體是什麼

如果你之後還是想加兩條額外虛擬線，做法是安裝 VB-Audio 的虛擬音訊驅動，不是接實體線。

具體概念如下：

1. 安裝 `VB-CABLE` 後，Windows 會多出一組 `CABLE Input / CABLE Output`。
2. 安裝 `VB-CABLE A+B` 後，Windows 會再多出 `CABLE-A` 與 `CABLE-B` 兩組虛擬裝置。
3. 每一組都像是一條「虛擬導線」：
   `Playback Device -> Recording Device`
4. 程式把聲音送進 `Input`，另一個程式就能從對應的 `Output` 收到。

但你的當前版本，我不建議第一步就裝到這麼複雜。先用 Banana 內建的兩組虛擬 I/O 最合理。

## 5. 專案目標

### 5.1 主要目標

做出一套可在單一 Windows 電腦上運作的雙向口譯工具，讓你在任何可選擇音訊裝置的視訊會議軟體中：

1. 聽到對方英文時，可看到英文辨識文字，並聽到中文譯音。
2. 你說中文時，可讓對方在會議中聽到英文譯音。
3. 你可以自行控制只開字幕、只開收聽翻譯、只開發話翻譯或雙向都開。

### 5.2 這版不追求的東西

1. 不追求企業部署。
2. 不追求多人管理。
3. 不追求雲端後台。
4. 不追求法規文件。
5. 不追求成本儀表板。
6. 不追求所有功能都做成 SaaS。
7. 不追求跟平台 API 深度整合。

### 5.3 成功標準

對你自己來說，這版只要達成以下條件就算成功：

1. 在至少 3 種不同會議軟體上能正常接入。
2. 對方英文能穩定變成英文字幕與中文語音。
3. 你的中文能穩定變成英文語音送到會議中。
4. 雙向同時開啟時不會嚴重回授或炸音。
5. 遇到裝置切換後，不需要重開整套程式才能恢復。

## 6. 建議的系統形態

### 6.1 不再使用瀏覽器擴充為主架構

既然目標是所有會議軟體，最合理的主程式形態是：

`Windows 本地桌面程式`

理由如下：

1. 會議軟體不一定都在瀏覽器中運作。
2. 你需要直接控制本機音訊裝置。
3. 你需要一個常駐的控制台來切換路由與模式。
4. 你需要隨時查看當前 bus 映射與音訊峰值。

## 7. 功能範圍

### 7.1 必做功能

1. 列出所有音訊輸入與輸出裝置。
2. 手動指定 `remote_in`、`local_mic_in`、`local_tts_out`、`meeting_tts_out`。
3. 接收對方語音並做英文辨識。
4. 顯示英文原文辨識字。
5. 產生中文翻譯。
6. 播放中文譯後語音到耳機。
7. 接收你的中文麥克風並做辨識。
8. 產生英文翻譯。
9. 輸出英文 TTS 到視訊會議軟體的虛擬麥克風。
10. 提供開始、停止、靜音、重置路由功能。

### 7.2 建議第二階段功能

1. 部分字幕與最終字幕。
2. 回溯最近 30 秒英文辨識內容。
3. Banana preset 一鍵套用。
4. 一鍵切換不同會議軟體設定檔。
5. 一鍵切換單向/雙向模式。
6. 短句模式與完整句模式切換。
7. 裝置斷線後自動重綁。

### 7.3 可延後功能

1. 本地離線模型全套部署。
2. 說話者分離。
3. 自動偵測多語混講。
4. 字幕 OCR 疊到其他程式視窗上方的美化效果。
5. 熱鍵全系統控制。

## 8. 雙向音訊流程設計

### 8.1 接收翻譯流程

這條是「對方講英文，你聽中文」。

#### 流程

1. 視訊會議軟體把對方聲音輸出到指定虛擬喇叭。
2. 翻譯程式從該虛擬輸出對應的錄音端擷取音訊。
3. 送進 ASR 做英文辨識。
4. 同步顯示英文原文。
5. 送進翻譯模組轉成中文。
6. 送進 TTS 轉成中文語音。
7. 輸出到 `local_tts_out`，預設為你的實體耳機。

#### 你要看見的內容

1. 英文原文辨識字。
2. 中文翻譯字。
3. 是否為 partial 或 final。
4. 當前延遲。

### 8.2 發話翻譯流程

這條是「你講中文，對方聽英文」。

#### 流程

1. 翻譯程式擷取你的實體麥克風。
2. 做中文 ASR。
3. 顯示中文辨識文字。
4. 翻譯成英文。
5. 英文文字送入 TTS。
6. TTS 輸出到虛擬麥克風。
7. 視訊會議軟體把該虛擬麥克風當成你的輸入。

#### 關鍵原則

1. 這條路徑不應再混入對方原音。
2. 如果你只想送英文譯音，就不要把實體麥克風直通到會議。
3. 若要保留原聲備援，應做成可切換，不可預設混在一起。

### 8.3 全雙工模式

全雙工模式就是兩條 pipeline 同時運作。

要成立，必須符合以下條件：

1. `remote_in` 與 `meeting_tts_out` 完全隔離。
2. 你的耳機監聽輸出不可被重新擷取成 `remote_in`。
3. 英文 TTS 不可被重新送回辨識對方語音的擷取端。
4. 中文 TTS 不可被送給會議軟體的麥克風輸入。

## 9. VoiceMeeter 映射範例

### 9.1 方案一：Banana-only，最推薦

這是目前最適合你的方案，不需要額外硬體，也不必先安裝額外 VB-CABLE。

#### 會議軟體設定

| 設定 | 裝置 |
| --- | --- |
| Speaker | `VoiceMeeter Input (VB-Audio VoiceMeeter VAIO)` |
| Microphone | `VoiceMeeter Output (VB-Audio VoiceMeeter VAIO)` |

#### 翻譯程式設定

| 邏輯通道 | 裝置 |
| --- | --- |
| `remote_in` | `VoiceMeeter Aux Output (VB-Audio VoiceMeeter AUX VAIO)` |
| `local_mic_in` | 你的 USB 麥克風或耳麥麥克風 |
| `local_tts_out` | 你的實體耳機 |
| `meeting_tts_out` | `VoiceMeeter AUX Input (VB-Audio VoiceMeeter AUX VAIO)` |

#### Banana 內部分工

| 條帶 | 作用 | 要開的路由 |
| --- | --- | --- |
| `VoiceMeeter Input` strip | 接收會議軟體播放的對方聲音 | `B2` only |
| `VoiceMeeter AUX Input` strip | 接收翻譯程式送出的英文 TTS | `B1` only |
| `A1` | 實體耳機 | 僅供中文 TTS 或必要測試 |
| Hardware Input 1 | 可留空或做麥克風監看 | 預設全關 |

#### 關鍵規則

1. `VoiceMeeter Input` strip 不要開 `A1`，因為你不需要聽對方原音。
2. `VoiceMeeter AUX Input` strip 不要開 `A1`，除非你想偷聽自己送出去的英文。
3. 麥克風建議讓翻譯程式直接讀，不要先進 Banana 再混。
4. 中文 TTS 建議由翻譯程式直接輸出到耳機，不佔 Banana 的第二條 virtual strip。

### 9.2 方案二：Banana + 額外 VB-CABLE，作為進階擴充

若之後你要做更細的隔離，可以再加 VB-CABLE，但不是現在的必要條件。

#### 適合加裝的情況

1. 想把中文 TTS 也納入 VoiceMeeter 控制。
2. 想要保留原音預覽。
3. 想讓會議音、譯音、系統音全部分開。

#### 一個可行的進階映射

| 用途 | 裝置 |
| --- | --- |
| 會議軟體 Speaker | `CABLE-A Input` |
| 翻譯程式 `remote_in` | `CABLE-A Output` |
| 翻譯程式 `meeting_tts_out` | `CABLE-B Input` |
| 會議軟體 Microphone | `CABLE-B Output` |
| 中文 TTS 監聽 | `A1` 或另一條虛擬通道 |

### 9.3 具體 bring-up 步驟

以下是你現在就可以照做的最小設定流程：

1. 啟動 VoiceMeeter Banana。
2. 把 `A1` 設成你的耳機，不要先用外放喇叭。
3. 將 `VoiceMeeter Input` strip 上只打開 `B2`。
4. 將 `VoiceMeeter AUX Input` strip 上只打開 `B1`。
5. 確認 `VoiceMeeter Input` 與 `AUX Input` 這兩條 strip 都沒有送到 `A1`。
6. 在會議軟體中把 Speaker 設成 `VoiceMeeter Input`。
7. 在會議軟體中把 Microphone 設成 `VoiceMeeter Output (B1)`。
8. 在你的翻譯程式中把 `remote_in` 設成 `VoiceMeeter Aux Output (B2)`。
9. 在你的翻譯程式中把 `meeting_tts_out` 設成 `VoiceMeeter AUX Input`。
10. 在你的翻譯程式中把 `local_tts_out` 設成實體耳機。
11. 先用錄音程式測 `remote_in` 是否收到對方聲音。
12. 再用測試音把英文 TTS 打進 `meeting_tts_out`，確認會議軟體麥克風電平有動。

## 10. 程式內部模組設計

### 10.1 `device_manager`

職責：

1. 列出所有 WASAPI 裝置。
2. 區分 input / output / loopback / virtual device。
3. 顯示裝置名稱、sample rate、channel 數、預設狀態。
4. 儲存使用者選擇。

### 10.2 `router`

職責：

1. 將邏輯通道映射到實際裝置。
2. 建立與停止音訊流。
3. 防止明顯錯誤映射。
4. 檢查是否有高風險回授。

例如：

1. `remote_in` 與 `meeting_tts_out` 指到同一個裝置時，拒絕啟動。
2. `local_tts_out` 指到正在被 `remote_in` 擷取的同一路時，提出警告。

### 10.3 `pipeline_remote`

職責：

1. 接收 `remote_in` 音訊。
2. 做英文 ASR。
3. 輸出英文文字。
4. 翻譯成中文。
5. 呼叫中文 TTS。
6. 輸出到 `local_tts_out`。

### 10.4 `pipeline_local`

職責：

1. 接收 `local_mic_in` 音訊。
2. 做中文 ASR。
3. 輸出中文文字。
4. 翻譯成英文。
5. 呼叫英文 TTS。
6. 輸出到 `meeting_tts_out`。

### 10.5 `transcript_buffer`

職責：

1. 暫存最近的辨識文字。
2. 保存原文與譯文對應。
3. 區分 partial / final。
4. 提供 UI 顯示與回溯。

### 10.6 `overlay_window`

職責：

1. 顯示對方英文原文。
2. 顯示對應中文翻譯。
3. 顯示你自己講的中文與英文結果。
4. 顯示當前音訊裝置與狀態。
5. 顯示音量峰值與延遲資訊。

### 10.7 `session_controller`

職責：

1. 控制開始與停止。
2. 切換單向 / 雙向模式。
3. 切換不同會議軟體預設。
4. 重置音訊設備。

   `Banana-only`

3. 一鍵套用推薦路由。

## 12. 分步驟實作計畫

這份計畫不再用商業專案排程思維，而是以「從最容易驗證到最難」的順序推進。

### Stage 0：先把路由模型想清楚

#### 目標

確定你最終採用的是哪種拓樸。

#### 工作

1. 安裝 VoiceMeeter Banana。
2. 列出所有 Windows 裝置名稱。
3. 先決定採用 Banana-only 最小拓樸。
4. 手動在不寫程式的情況下先測通音訊路由。
5. 額外 VB-CABLE 先視為可選，不列為第一步必要安裝。

#### 完成標準

1. 你能在 Windows 裡看到所有必要裝置。
2. 你能用錄音程式驗證 `remote_in` 與 `meeting_tts_out` 對應的路徑是可用的。
3. 你已知道哪個 bus 負責哪件事。

### Stage 1：只做接收端擷取

#### 目標

先證明「會議軟體的對方聲音」能被你的程式穩定抓到。

#### 工作

1. 建立裝置列舉功能。
2. 建立從指定 input device 持續讀取音訊的程式。
3. 加入音量波形或音量條。
4. 測試 Zoom、Teams、Meet 至少其中兩種。

#### 完成標準

1. 切換不同會議軟體時，`remote_in` 都能抓到聲音。
2. 你的耳機監聽不會導致再次被擷取。

### Stage 2：只做發話端擷取

#### 目標

先證明你自己的麥克風能穩定輸入程式。

#### 工作

1. 讀取實體麥克風。
2. 顯示麥克風音量。
3. 加入靜音與增益控制。
4. 加入裝置切換。

#### 完成標準

1. 麥克風斷線或切換時可以恢復。
2. 你的講話不會誤送到 `remote_in`。

### Stage 3：完成接收翻譯鏈

#### 目標

完成「對方英文 -> 英文字幕 -> 中文翻譯 -> 中文 TTS -> 你耳機」。

#### 工作

1. 為 `remote_in` 建立 ASR worker。
2. 顯示英文原文。
3. 接上翻譯模組。
4. 顯示中文譯文。
5. 接上中文 TTS。
6. 把 TTS 播放到 `local_tts_out`。

#### 完成標準

1. 你能聽見中文譯音。
2. 你能看見英文辨識字。
3. 原文與譯文能對得上。

### Stage 4：完成發話翻譯鏈

#### 目標

完成「你中文 -> 中文辨識 -> 英文翻譯 -> 英文 TTS -> 會議麥克風」。

#### 工作

1. 為 `local_mic_in` 建立 ASR worker。
2. 顯示中文辨識字。
3. 接上英文翻譯。
4. 接上英文 TTS。
5. 把英文 TTS 播放到 `meeting_tts_out`。
6. 用錄音軟體驗證該虛擬麥克風有正常輸出。

#### 完成標準

1. 會議軟體能從 Banana 的 `B1` 正常吃到英文譯音。
2. 對方能聽到英文譯音。
3. 你的原始中文不會一起送出去，除非你主動開啟直通。

### Stage 5：整合成雙向模式

#### 目標

讓兩條 pipeline 同時開啟。

#### 工作

1. 同步啟動 `pipeline_remote` 與 `pipeline_local`。
2. 為兩條路徑分別顯示狀態。
3. 加入 bus 衝突檢查。
4. 加入回授警示。
5. 加入一鍵重置路由。

#### 完成標準

1. 雙向同時運作。
2. 不會因為英文 TTS 被 `remote_in` 再抓回而無限循環。
3. 不會因為中文 TTS 被送進會議麥克風路徑而亂掉。

### Stage 6：加入字幕與控制面板

#### 目標

讓這套東西真正能在會議中順手使用。

#### 工作

1. 製作簡潔主視窗。
2. 顯示四欄字幕。
3. 顯示 partial / final 狀態。
4. 加入開始、停止、靜音、切模式按鈕。
5. 加入裝置測試按鈕。

#### 完成標準

1. 不看 console 也能操作。
2. 能看出是哪條路徑出錯。

### Stage 7：優化穩定性與延遲

#### 目標

把它從「能跑」調整成「真的能用」。

#### 工作

1. 調整 chunk size。
2. 調整 ASR 分段策略。
3. 決定短句快譯或整句穩譯模式。
4. 加入耳機監聽音量管理。
5. 降低中文 TTS 與你自己說話時的互相干擾。
6. 加入最近 30 秒回溯。

#### 完成標準

1. 會議中不覺得嚴重卡頓。
2. 錯路由時能快速修復。

## 13. 最小可行版本建議

最務實的 MVP 不是直接做雙向全雙工，而是分三段：

### MVP-1

只做：

1. 對方英文辨識。
2. 顯示英文文字。
3. 中文 TTS 給你自己聽。

### MVP-2

再加：

1. 你的中文辨識。
2. 英文翻譯。
3. 英文 TTS 送到虛擬麥克風。

### MVP-3

最後才加：

1. 雙向同開。
2. 回授檢查。
3. 原音 ducking。
4. 回溯與 partial/final。

這樣做的原因很簡單：

1. 最常見的問題不在模型，而在音訊路由。
2. 如果一開始就雙向同開，除錯會非常混亂。
3. 單向路徑先跑通，後面只是把兩條獨立可用的鏈接起來。

## 14. 音訊與模型層的具體建議

### 14.1 ASR 建議

如果你想先求成功率與開發速度：

1. 先選一個品質穩定的現成 ASR。
2. 先只做英文與中文。
3. 先做串流或偽串流，不必一開始就追超低延遲。

### 14.2 翻譯建議

翻譯層只要滿足兩件事就夠：

1. 短句能快出。
2. 完整句能修正前面的 partial。

### 14.3 TTS 建議

TTS 在這個專案裡很重要，因為你要靠耳朵即時理解。

建議順序：

1. 先用穩定、容易接的 TTS。
2. 確認音訊輸出裝置可控。
3. 再追求自然度與串流化。

### 14.4 不要太早做的事

1. 不要太早做術語庫。
2. 不要太早做多語自動偵測。
3. 不要太早做本地大型模型。
4. 不要太早做過度複雜的 UI。

## 16. 風險與對策

這版不談商業法規，但仍有技術風險。

| 風險 | 說明 | 對策 |
| --- | --- | --- |
| 回授循環 | 英文 TTS 被再次抓回 `remote_in` | 嚴格拆分 bus，啟動前做衝突檢查 |
| 裝置名稱變動 | Windows 重新排序裝置 | 用裝置 GUID 或名稱比對 + 重綁 |
| 取樣率不一致 | 不同裝置 44.1k / 48k 不同 | 進程內統一重取樣 |
| 會議軟體獨佔裝置 | 某些軟體鎖裝置 | 優先用虛擬裝置，避免搶實體設備 |
| TTS 太慢 | 翻譯出來但播音落後 | 先做字幕，再優化 TTS chunk |
| 雙向一起開太亂 | 除錯困難 | 分段開發，先單向後雙向 |

## 19. 最終建議的開發順序

如果你現在就要開始做，最好的順序是：

1. 先把 VoiceMeeter Banana 手動路由測通。
2. 寫一個只顯示裝置與音量的測試程式。
3. 寫 `remote_in -> 英文字幕`。
4. 再加 `remote_in -> 中文 TTS`。
5. 寫 `local_mic_in -> 中文字幕 -> 英文 TTS -> meeting_tts_out`。
6. 最後再整合雙向同時運作。

這個順序的好處是每一步都能單獨驗證，而且出問題時你知道是：

1. 音訊路由問題。
2. 模型問題。
3. TTS 輸出問題。
4. 會議軟體裝置設定問題。

---

## 20. 結論

若目標是「只有你自己使用，但要盡量支援所有視訊會議軟體」，那麼放棄平台整合、改走音訊整合，是正確方向。

而且在這個方向下，最重要的不是先寫翻譯模型，而是先把以下四件事做穩：

1. `remote_in` 要穩定抓到對方聲音。
2. `meeting_tts_out` 要穩定送到 Banana 的送話路徑。
3. `local_tts_out` 要只給你自己聽，不要污染其他路徑。
4. 所有 bus 的回授關係要先在架構上切乾淨。

因此本版計畫的總結是：

1. 以 Windows 桌面程式為主。
2. 以虛擬音訊裝置整合為主。
3. 以邏輯通道而非平台 API 為主。
4. 以單向測通再雙向整合的順序開發。
5. 第一版優先採用「VoiceMeeter Banana 內建兩組虛擬 I/O」的拓樸。

這會比直接綁死在 Google Meet、Teams 或純瀏覽器擴充方案更接近你真正要的結果。
